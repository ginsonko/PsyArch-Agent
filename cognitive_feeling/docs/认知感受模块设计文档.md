# 认知感受模块（CFS）设计文档（模块 5）

生成时间：2026-04-12  
理论依据：`txt版本的理论核心.txt`（重点参考：3.3.1、3.8、3.9、3.10、3.12）

> CFS（Cognitive Feeling Signals, 认知感受信号）不是“拟人贴标签”，而是 AP 的**自我调控与自我认知**信息通道：既能调制注意/学习/行动，又能作为可被匹配、可被学习的对象进入系统。

---

## 1. 模块定位与职责边界

### 1.1 核心定位（理论 3.8）
CFS 在 AP 中至少承担三类作用：
1. **功能性评价**：评价当前认知是否顺畅、是否存在未解释残差、是否需要继续展开或收敛停止。
2. **调制输入**：作为 AF（注意力）、学习强度、行动阈值的重要调制信号。
3. **元认知信息载体**：CFS 本身也是信息（通常以刺激元形式存在），可写入 SP 参与后续匹配、联想、注意力与行动决策。

### 1.2 工程职责
认知感受模块要做的不是“生成自然语言情绪”，而是：
- 从当前 tick 的关键可观测变量与事件中（`E_r/E_v/Δ/P`、匹配得分、残差、赋能稳定性、奖励/惩罚信号、路径疲劳等）**生成一组结构化 CFS 信号**；
- 将 CFS 信号以“属性刺激元”形式绑定到对应对象（通常绑定到某个 ST/SA 的 CSA 上），或作为全局信号写入状态池；
- 输出可审计的触发原因与强度，供 observatory 展示与后续脚本调参。

### 1.3 不做什么
- 不直接改变 HDB 权重或写库（这些是 HDB/EMgr/IESM 的职责）
- 不决策最终行动（行动由 Drive 竞争与行动器执行）
- 不把 CFS 当作静态标签（必须与能量/匹配事件/趋势等变量绑定）

---

## 2. 表达形式：刺激元化 + CSA 绑定（理论 3.8.1）

### 2.1 为什么必须“刺激元化”
理论建议将大多数认知感受实现为**属性刺激元**，并以 CSA 形式绑定到对应对象，例如：
- `【结构X；违和感:高；来源:内源；能量种类:虚】`
- `【结构Y；期待:中；奖励信号:存在】`

这样做的直接好处：
1. CFS 可被查存一体自然学习（与对象共现，未来能被抽象）
2. CSA 完全匹配规则避免跨对象误触发（不会把 A 的违和感匹配到 B 上）

### 2.2 两类 CFS
- **对象绑定型 CFS**：绑定到具体 ST/SA（更常见）
- **全局型 CFS**：不绑定对象，例如“繁/简”“无聊”等，可作为独立特征刺激元或绑定到“当前状态对象”

## 3. 关键感受的逻辑约束（理论 3.8.2，必须对齐）

下面给出理论中明确的“硬约束”，实现时必须保持：

### 3.1 违和感
触发条件（任一满足即可）：
- 预测（虚能量）与现实（实能量）在关键对象上不一致
- 或认知压 `P` 显著升高且无法通过匹配降低

语义：需要解释/需要纠错。

### 3.2 正确感（正确事件）
硬约束：
- 正确感必须依托于“违和感的消失/显著下降”产生
- 若从始至终没有违和，则不产生正确感（可处于中性无感）

### 3.3 惊（惊讶/惊异）
触发：连续或强烈的意外输入（实能量侧突增导致的短时高偏差）。
作用：驱动警觉、注意力切换、探索性核对。

### 3.4 期待
硬约束：
- 期待来源于**预测对象（虚能量对象）内部包含奖励信号成分**
- 随该预测结构的 `E_v` 上升而增强，`E_v` 下降则期待下降
- 当包含期待属性的更大结构在匹配中获得实能量（现实验证发生）时，形成“期待被证实体验”，可触发更高阶感受并驱动情绪与权重更新

### 3.5 压力
与期待对称：
- 来源于预测结构内部包含惩罚信号成分
- 随 `E_v` 上升而增强，验证发生时形成“压力被证实体验”

## 4. 可扩展通道示例（理论 3.8.3）

IESM 脚本可扩展的感受示例包括：
- 情绪状态感受（来自 NT 通道阈值）
- 新鲜感（基于 HDB 近因增益/更新时间）
- 重复感（基于状态池疲劳）
- 繁/简（当前注意图景复杂度）：由 CAM 大小与能量集中度生成
- 把握感/置信度（虚能量稳定性）：由虚能量分布稳定程度、关键结构的 EV 覆盖程度、以及认知压下降趋势生成

并支持感受之间的融合、抑制与二阶触发（3.8.4），例如：
- “繁 + 惩罚信号 → 烦躁”
- “正确事件抑制违和”

## 5. 接口设计（对齐原型阶段）

### 5.1 输入（建议）
MVP 最小输入建议为三类（都能在现有实现中观测到）：
- `cam_snapshot`（来自注意力模块）
- `structure_level_result`（来自结构级查存一体：ρ_k、残差、偏置赋能等摘要）
- `state_pool_snapshot`（来自状态池：fatigue/recency/ΔP 等动态）

### 5.2 输出（建议）
输出两条路线：
1. `cfs_signals` 列表（每个信号包含：kind/strength/bind_target/trigger_reasons）
2. `state_pool_patch`（把 CFS 作为属性 SA/CSA 绑定写回状态池的操作建议，实际写入可由状态池模块执行）

### 5.3 预留接口
- 与 IESM 脚本系统对接：阈值、融合/抑制规则、脚本版本审计
- 与 EMgr 对接：把“正确事件/期待/压力/惊/违和持续”映射到 NT 更新输入
- 与 Action/Drive 对接：把“违和/惊/繁”映射为注意力聚焦行动器触发条件

## 6. 现阶段可接入观测循环的部分

在模块 4（注意力）单独落地后，模块 5 可以先实现一个 **只读观测版 MVP**：
- 不写回状态池（先只生成并展示 CFS 诊断）
- 先实现最容易严格对齐的信号：
  - 违和趋势（P 高且多轮不降）
  - 正确事件（违和显著下降）
  - 繁/简（CAM 尺寸与集中度）
  - 重复感（fatigue 高）

等 IESM/EMgr/Action 完成后，再把写回与联动打开。
