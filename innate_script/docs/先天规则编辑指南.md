# 先天规则编辑指南（先天编码脚本管理器 IESM）

本指南对应原型阶段的 **先天编码脚本管理模块（IESM, Innate Encoded Script Manager）** 的“声明式规则系统”，用于把关键触发逻辑外置到 YAML 文件，并在观测台前端提供可编辑入口。

为了可读性：本项目尽量用中文说明；出现英文缩写时，会同时给出中文全称。

---

## 0. 术语与缩写（Glossary）

- 先天编码脚本管理器（IESM, Innate Encoded Script Manager）
- 认知感受系统（CFS, Cognitive Feeling System）
- 注意力过滤器（AF, Attention Filter）
- 状态池（SP, State Pool）
- 组合刺激元（CSA, Combined Stimulus Atom）
- tick（时间步）：原型的基本循环计数单位

---

## 1. 规则文件在哪里？

- 默认规则文件：`innate_script/config/innate_rules.yaml`
- 自动备份目录：`innate_script/config/rules_history/`
  - 每次“保存并热加载”都会先备份旧文件，再写入新文件，便于回滚。

> 如果你在 `innate_script/config/innate_script_config.yaml` 里设置了 `rules_path / rules_backup_dir`，则会使用你指定的路径。

---

## 2. 如何在前端编辑？

1. 启动观测台：运行 `python -m observatory`（或双击 `快速启动观测台.bat`）。
2. 在浏览器侧边栏进入 **「先天规则」** 页面。
3. 推荐流程：
   - 先用左侧表单新增/编辑规则
   - 点击 **「校验」**（会生成规范化 YAML 预览 + 错误/警告列表）
   - 点击 **「保存并热加载」**
4. 点击 **「用最近一轮模拟」**
   - 会用最近一次 cycle 的上下文进行 dry-run（不修改冷却记账），用于快速验证规则是否触发、会输出哪些指令。

### 2.1 图形编排（B目标：可视化）

如果你希望像智能家居规则、游戏节点编辑器那样拖拽连线：

1. 在规则详情页点击 **「图形编排（可视化）」** 按钮。
2. 在弹窗里：
   - 添加条件节点（认知感受 / 状态窗口 / 定时器）
   - 添加动作节点（聚焦指令 / 触发记录 / 日志）
   - 用连线把「条件节点 -> 条件汇总 -> 动作节点」串起来
   - 视图操作：
     - 缩放：顶部滑块、`100%`、`适配`；或 `Ctrl/⌘ + 滚轮`
     - 平移：按住空格键再拖拽空白处
     - 滚动：画布支持上下/左右滚动条（节点跑出视野也能找到）
3. 点击 **「应用到规则」** 把图形编排结果写回到表单草稿。
4. 回到规则页继续 **校验 / 保存并热加载**。

> 重要：图形编辑器只是“编辑方式”，最终仍然会编译回 `when/then` 的 YAML；不会执行任意代码。

## 3. 规则结构概览（Schema）

文件顶层（Top-level）：

- `rules_schema_version`：结构版本（当前 `"1.0"`）
- `rules_version`：规则版本号（用于审计/回滚追踪）
- `enabled`：总开关
- `defaults`：动作默认参数（动作里未填写时使用）
- `rules`：规则列表

每条规则（Rule item）：

- `id`：唯一标识（建议稳定，便于审计与引用）
- `title`：标题（显示用）
- `enabled`：启用开关
- `priority`：优先级（越大越优先；冲突仲裁预留）
- `cooldown_ticks`：冷却 tick（避免每 tick 重复触发）
- `when`：触发条件（支持 `any/all/not` + 叶子谓词）
- `then`：动作列表
- `note`：备注（给人看的注释）

## 4. when：支持的典型条件（触发条件）

### 4.1 CFS 信号触发（cfs）

```yaml
when:
  cfs:
    kinds: ["dissonance", "pressure"]
    min_strength: 0.3
```

含义：只要本 tick 的 `cfs_signals` 里出现上述 kind，且强度 >= 阈值，就认为命中。

### 4.2 状态窗口触发（state_window）

```yaml
when:
  state_window:
    stage: "any"            # any | maintenance | pool_apply
    fast_cp_rise_min: 1
```

含义：任一状态窗口（维护窗口/落地窗口）满足认知压快速上升计数 >= 阈值就命中。

### 4.3 定时触发（timer）

```yaml
when:
  timer:
    every_n_ticks: 10
```

含义：每 10 个 tick 触发一次（tick_index 来自 `cycle_0001` 末尾数字或观测台 tick_counter）。

### 4.4 指标条件触发（metric，推荐用于“能量/变化率/奖励惩罚/递质/匹配分数”等）

`metric` 是原型阶段最通用的条件类型，用于表达你提到的典型条件：

- `__获得实能量__ / __获得虚能量__ / __认知压状态__ / __认知压变化率__`
- `奖励信号/惩罚信号 状态/增加/变化率`
- `状态池 ER/EV/CP 总量、对象数量、能量聚集度（以及这些指标的变化率）`
- `刺激级剩余能量比例`
- `查存一体过程匹配分数（支持按目标 structure_id 选择）`
- `情绪递质状态/变化了/变化率（推荐用 preset=nt_state/nt_delta/nt_rate/nt_changed + channel；也可手写 emotion.nt.<通道>）`

示例 1：任一对象“获得实能量（ER）”大于 0（可选过滤：只看包含某特征的对象）

```yaml
when:
  metric:
    preset: "__获得实能量__"   # 中文别名会在保存时归一化为 got_er
    mode: "delta"
    op: ">"
    value: 0
    selector:
      mode: "contains_text"
      contains_text: "你好"
```

示例 2：状态池能量聚集度过低（更“分散”，可能对应后续的“繁/简”感受）

```yaml
when:
  metric:
    preset: "pool_concentration"
    op: "<"
    value: 0.18
```

示例 2.5：状态池实能量变化率（ΣER）为正（近 4 tick 平均在上升）

```yaml
when:
  metric:
    preset: "pool_er_rate"
    window_ticks: 4
    op: ">"
    value: 0
```

示例 3：某个情绪递质通道“变化了”

```yaml
when:
  metric:
    preset: "nt_changed"
    channel: "多巴胺"            # 也可写 "DA"（两者都支持）
    op: "changed"
    mode: "delta"
    epsilon: 1e-6
```

示例 4：查存一体匹配分数（按目标 structure_id）

```yaml
when:
  metric:
    preset: "stimulus_match_score_target"
    op: ">="
    value: 0.60
    selector:
      mode: "specific_ref"
      ref_object_id: "st_000012"   # 目标 structure_id
```

关键字段说明：

- `preset`：指标预设名（推荐优先用预设；UI 下拉框也会列出）
  - 支持中文别名，例如 `__获得实能量__` / `__认知压变化率__` / `奖励信号状态__` 等
  - 保存/校验阶段会归一化到稳定英文键（例如 `got_er`）
- `metric`：指标路径（高级写法；未填 preset 时使用）
  - 典型：`item.er` / `pool.total_er` / `emotion.rwd` / `emotion.nt.DA` / `retrieval.stimulus.match_scores`
- `mode`：取值方式
  - `state`：当前值
  - `delta`：近 1 tick 变化量（需要 tick_index；观测台 run_cycle 会提供）
  - `avg_rate`：近 N tick 平均变化率（窗口长度 `window_ticks`，建议 3~4）
- `op`：比较符：`>= > <= < == != between exists changed`
- `selector`：对象选择器（用于 `item.*` 指标；也可用于 `match_scores` 的目标选择）
  - `mode: all / specific_ref / specific_item / contains_text / top_n`
  - `contains_text` 会匹配：display / 详情 / 属性 / 特征 / 运行态绑定属性展示
- `capture_as`：变量捕获（可选）
  - 命中后会把数值写入变量，例如 `capture_as: gain` -> 可在动作里使用 `{{{gain}}}`
  - 若命中的是 `item.*` 指标，还会额外提供：`{{{gain_item_id}}}` / `{{{gain_ref_object_id}}}` / `{{{gain_ref_object_type}}}`，便于把后续动作精确绑定到同一对象
  - 若命中的是刺激级查存一体匹配分数（retrieval.*），会尽力提供 `{{{gain_ref_object_id}}}`（通常为 best_match_target_id / structure_id）

### 4.4 组合条件（any / all）

任一触发（OR）：

```yaml
when:
  any:
    - cfs: { kinds: ["surprise"], min_strength: 0.4 }
    - state_window: { fast_cp_rise_min: 3 }
```

同时满足（AND）：

```yaml
when:
  all:
    - cfs: { kinds: ["dissonance"], min_strength: 0.3 }
    - state_window: { stage: "maintenance", fast_cp_rise_min: 1 }
```

## 5. then：支持的动作（执行动作）

### 5.0 cfs_emit：生成认知感受信号（CFS，可选绑定为属性刺激元）

`cfs_emit` 会把一条“认知感受信号（CFS, Cognitive Feeling System signal）”追加到本 tick 的运行态列表中，供：

- 后续同 tick 的 IESM 规则继续消费（例如触发行动）
- 情绪管理器（EMgr/NT）与行动模块（Action/Drive）在同 tick 内消费
- 观测台展示与审计

典型写法：

```yaml
then:
  - cfs_emit:
      kind: dissonance
      from: metric_matches
      scope: object
      max_signals: 8
      strength:
        from: match_value
        policy: linear_clamp
        min: 0.5
        max: 2.0
        out_min: 0.3
        out_max: 1.0
      bind_attribute:
        attribute_name: cfs_dissonance
        display: "违和感:{{{strength}}}"
        value_from: strength
        value_type: numerical
        modality: internal
        er: 0.0
        ev: 0.0
```

#### strength.policy：支持的映射策略

- `linear_clamp`：线性钳制映射（最常用）
- `scale_offset`：比例偏移（简单线性变换）
- `verify_mix`：渐变验证混合（用于“期待/压力”的验证与不验，不是二极管）

#### verify_mix：数学定义（连续、对称、可部分验证）

目的：把“验证/不验”从二选一改成连续渐变，且两者对抗并对称。

输入（两股对抗量）：

- `pred`：预测量（通常取 EV，或其映射后的强度）
- `actual`：实际量（通常取 got_er / er_rate 等）

混合占比（默认 gamma=1）：

- `v = actual / (actual + pred + eps)`
- `u = pred / (actual + pred + eps)`

输出强度：

- `strength_verified = base_strength * v`
- `strength_unverified = base_strength * u`

其中：

- `base_strength` 通常来自上一条 `expectation/pressure` 的强度（用 `capture_as` 捕获）
- `eps` 用于避免 0 除（默认 `1e-6`）
- `gamma` 可调“果断度”：>1 更接近二极管，<1 更柔和（建议 0.7~2.0）
- `pred_scale/actual_scale` 用于量纲对齐（常见：`actual_scale=window_ticks`）

示例：期待在 2 tick 后做渐变验证（两条信号对抗）

```yaml
- cfs_emit:
    kind: expectation
    from: metric_matches
    scope: object
    max_signals: 1
    capture_as: expect_strength
    strength: { from: match_value, policy: linear_clamp, min: 0.3, max: 1.5, out_min: 0.3, out_max: 1.0 }
- delay:
    ticks: 2
    then:
      - branch:
          when:
            metric:
              preset: er_rate
              window_ticks: 2
              selector: { mode: specific_item, item_id: "{{{match_item_id}}}" }
              op: exists
              capture_as: expect_er_rate
          then:
            - cfs_emit:
                kind: expectation_verified
                from: metric_matches
                scope: object
                strength:
                  from: var
                  var: expect_strength
                  policy: verify_mix
                  part: verified
                  pred_var: expect_pred_ev
                  actual_var: expect_er_rate
                  actual_scale: 2.0
            - cfs_emit:
                kind: expectation_unverified
                from: metric_matches
                scope: object
                strength:
                  from: var
                  var: expect_strength
                  policy: verify_mix
                  part: unverified
                  pred_var: expect_pred_ev
                  actual_var: expect_er_rate
                  actual_scale: 2.0
```

> 注意：verify_mix 只做“连续对抗信号”的生成，本身不执行学习；预测收敛由 HDB 的权重机制以及后续学习模块负责。

### 5.1 focus：输出注意力聚焦指令（下一 tick 生效）

```yaml
then:
  - focus:
      from: "cfs_matches"          # cfs_matches | state_window_candidates
      match_policy: "all"          # all | strongest | first
      ttl_ticks: 2
      focus_boost: 0.9
      deduplicate_by: "target_ref_object_id"
```

效果：

- 当行动模块（Action/Drive）启用时：IESM 输出的 `focus_directives` 会作为 **行动触发源** 输入行动模块；行动模块在本 tick 内做 Drive（驱动力）竞争与消耗，若执行成功，会输出 `focus_directives_out`，并在 **下一 tick** 生效。
- 当行动模块禁用时：观测台会把 `focus_directives` 直接加入 pending，并在 **下一 tick** 输入给注意力过滤器（AF），用于加成排序。

### 5.2 emit_script：输出“触发记录”（用于观测/联调）

```yaml
then:
  - emit_script:
      script_id: "innate_state_window_cp_rise"
      script_kind: "window_trigger"
      priority: 50
      trigger: "fast_cp_rise"
```

说明：原型阶段主要用于让你在观测台/报告中看到“某规则命中了”，后续可扩展为更复杂的指令链。

### 5.3 emotion_update：更新情绪递质通道（同一 tick 生效）

```yaml
then:
  - emotion_update:
      DA: 0.10
      COR: -0.05
```

含义：把通道增量交给情绪管理器（EMgr/NT）合并执行，在同一 tick 内影响 `nt_state_after` 与 `modulation`。

说明：通道缩写及其中文名映射在情绪模块里维护（例如 `DA=多巴胺`、`COR=皮质醇`）。

### 5.4 action_trigger：结构化行动触发（进入 Drive 竞争）

```yaml
then:
  - action_trigger:
      action_id: "custom_action"
      action_kind: "custom"
      gain: 0.3
      threshold: 1.0
      cooldown_ticks: 0
      params: {}
```

含义：把结构化触发作为“行动节点的驱动力增益”输入行动模块（Action/Drive）。

注意：原型阶段行动模块仅内置执行少数 `action_kind`（如 `attention_focus`/`attention_focus_mode`/`attention_diverge_mode`/`recall`）。未知 kind 会被保留为节点但不会执行（安全）。

### 5.5 pool_energy：对状态池对象赋能（ER/EV 增量）

```yaml
then:
  - pool_energy:
      selector:
        mode: "contains_text"
        contains_text: "你好"
      delta_er: 0.10
      delta_ev: -0.02
      reason: "demo_pool_energy"
```

含义：输出“状态池赋能效果（pool_effects）”，由观测台的安全执行器立即落地到状态池（SP）。

说明：

- 数值字段支持模板变量（例如 `delta_er: "{{{gain}}}"`）。
- 支持 `create_if_missing: true`（仅对 `specific_ref` 且增量为正时生效），用于“状态池不存在该对象时自动创建”。

### 5.6 pool_bind_attribute：创建属性刺激元并绑定到对象（折叠到锚点快照）

```yaml
then:
  - pool_bind_attribute:
      selector:
        mode: "specific_ref"
        ref_object_id: "st_000012"
        ref_object_type: "st"
      attribute:
        attribute_name: "confidence"
        attribute_value: 0.72
        raw: "confidence:0.72"
        display: "置信度:0.72"
        value_type: "numerical"
        modality: "internal"
        er: 0.01
        ev: 0.00
      reason: "demo_bind_confidence"
```

含义：创建一个运行态“属性刺激元（SA, role=attribute）”，并绑定到目标对象。

重要：为避免状态池内出现大量 `CSA[...]` / 属性 SA 的噪音对象，原型默认不把这些属性 SA 作为独立状态池项写入；它们会被折叠进目标对象的 `bound_attribute_displays` / `display_detail` 中。

### 5.7 delay：延时执行（tick 级调度）

```yaml
then:
  - delay:
      ticks: 2
      then:
        - log: "2 tick 后执行"
        - emit_script: { script_id: "after_delay" }
```

说明：delay 不会阻塞当前 tick；会把动作入队到运行态调度队列，到期后再执行。

### 5.8 branch：动作内分支（当条件满足/不满足/报错）

```yaml
then:
  - branch:
      when:
        metric: { preset: "reward_state", op: ">", value: 0 }
      then:
        - log: "满足条件"
      else:
        - log: "不满足"
      on_error:
        - log: "条件报错"
```

说明：branch 的 `when` 支持和顶层规则相同的 when 表达式（any/all/not/cfs/state_window/timer/metric）。

---

## 6. 已知限制（Prototype Limitations）

- 当前规则系统是 MVP：不支持任意代码执行，不支持复杂数学表达式（这是刻意的安全设计）。
- 图形编排（B目标）目前覆盖更多节点类型（含 `metric`、`emotion_update`、`pool_energy` 等）。但复杂动作（例如嵌套 then/branch）仍可能更适合在表单或 YAML 中编辑，后续可继续增强节点化体验。
- `action_trigger` 已接入主流程并进入 Drive 竞争；但未知 `action_kind` 不会被执行（会保留为节点，便于后续扩展行动器）。
- 更完整的“冲突仲裁、安全底线、权限沙箱、版本回滚 UI”会在后续迭代逐步增强。
