## 1. 摘要

认知拼接（Cognitive Stitching, CS）是一种在状态池维护之后、注意力筛选之前运行的“事件认知组织”机制。它利用 HDB 中每个结构的局部残差信息（共现残差与时序残差），在运行期动态形成并维护“事件结构”（Event Structure, ES），并将 ES 作为真正的结构对象放入状态池，与普通结构共同参与能量竞争、注意力入选与后续查存一体流程。

与传统的“结构级查存一体”相比，CS 的核心差异在于：
- **事件作为结构而不是关系**：事件结构可被注意力直接选取、可携带自身 ER/EV、可拥有自身数据库索引与残差表，并可在后续步骤中以“整体对象”被处理，从而提升拟人化的“注意到某件事”的体验。
- **资源纪律来自能量账本而非硬规则**：ES 的生成与扩展需要吸收组分的部分能量，形成近似守恒；同时以半衰期、淘汰阈值与容量软上限驱动自然的长度约束与事件竞争，而不是依赖大量 if/else 的手工禁令。
- **弱命中与动态顺序证据**：顺序敏感不再意味着“不能匹配”，而是通过连续化的弱命中机制决定匹配强度；内源性刺激（常弱顺序）依旧能以较低置信度命中顺序结构，从而符合“动态演化、允许中间态”的设计美学。
- **事件到事件的桥接拼接**：CS 不仅允许普通结构进入事件，也允许事件结构之间依据“后缀-前缀上下文”“部分序列”“弱顺序证据”进行桥接；强命中吸收更多能量，弱命中仅形成较弱整体，并通过短时抽能疲劳避免同一路径在短时间内反复抽干组分。
- **懒融合 + 闲时巩固**：ES 的数据库以“父链 + 增量”的形式懒融合，运行期只做 top-k 级别叠加；在空 tick/闲时通过“巩固”将父链压平、释放缓存与内存，防止数据库分叉长期堆积。
- **缓存中和联动到 ES 组分**：当外源性刺激提供了对 ES 内部某组分的真实证据时，允许对该组分的 ER/EV 失衡进行“互补中和”，只降低认知压、不抬高认知压，形成“整体更稳、碎片更可解释”的拟人动态。

工程上，CS 不直接删除既有结构级查存一体，而是：
- 增加结构级查存一体开关，默认关闭，用于 A/B、回滚与对照；
- 增加 CS 模块开关，默认开启；
- 补齐可观测字段、长期图表、调参规则与闲时巩固按钮，使其满足原型技术规范的可审计要求。

## 2. 问题背景与动机

在既有运行报告中，长期稳定出现的核心异常可以概括为：
- 结构级查存一体耗时长期超上限，出现多次秒级尖峰；
- 结构级轮次与候选空间过宽，导致 `attention_state_pool_candidate_count`、`structure_round_count`、`timing_structure_level_ms` 等指标共振；
- HDB 结构数量持续膨胀但“有效候选”未收敛，呈现“越跑越肥、越肥越慢”的负反馈；
- 时间感受等链路在复杂段放大，叠加上游候选过宽，进一步推高总耗时；
- 调参器容易误把结果指标当控制点（如直接打 detail budget），导致高回滚与试错震荡。

这些现象并不必然反证理论，但会掩盖理论希望凸显的优势：局部化、预算可控、可解释、低算力持续运行。因此需要一种新的组织方式，在不降低逻辑效果的前提下，把“事件认知”从高成本的结构级查存一体路径中分离出来，转移到一个更局部、更可控、可渐进演化的机制上。

认知拼接的动机来自一个观察：
- 历史经验的可迁移性，本质上并不是“记住所有细节”，而是“在相似的状态组合下，倾向于采取什么操作与回顾路径”。  
- 在既有系统里，外源刺激与内源刺激都进入刺激级查存一体并写入记忆；因此“状态池现状的信息”在刺激级层面确实会被存入。  
- 如果刺激级查存一体能直接命中“最大匹配结构”，似乎结构级查存一体的部分功能会被重复。  

然而，刺激级查存一体的目标更接近“对输入刺激包的结构化解释与残差切割”，它不天然等价于“把状态池中的结构对象组织成可被注意力整体把握的事件”。换言之，刺激级能形成结构，但不保证形成“运行期可竞争、可淘汰、可反复演化的事件对象”。  

CS 的目标是用更少的强制规则，把“事件对象”这件事交还给系统自身的能量与竞争机制，让“整体叙事”与“碎片解释”在同一套能量语义下自然切换。

### 2.1 两种“历史相似性”：刺激相似 vs 状态图景相似
在持续认知系统中，需要区分两类相似：
- 刺激相似：当前输入的刺激序列与历史刺激序列相似（文本片段、感知片段、工具返回片段等）。
- 状态图景相似：当前状态池中同时激活的一组结构及其能量分布，与历史某一时刻的“整体局面”相似（包含主次关系、压力/期待/违和背景等）。

刺激级查存一体天然擅长前者，因为它以“刺激包”为输入并进行最大共同部分切割；  
认知拼接与事件结构更贴近后者，因为它以“状态池快照”为输入，将离散结构组织成可竞争的整体事件，从而影响注意力与后续操作倾向。

## 3. 核心术语与对象定义（面向初学读者）

本节用最小假设解释本文档反复使用的对象与概念。读者不需要预先理解任何实现细节。

### 3.1 Tick 与状态池（State Pool, SP）
系统以离散 tick 运行。每个 tick 包含：
- 状态池维护（能量衰减、淘汰、合并等）
- 注意力筛选（从状态池中选取本 tick 的注意力记忆体 CAM）
- 查存一体（结构级、刺激级）
- 内源刺激化（把残差等转为内源刺激）
- 刺激级处理与后续感应赋能、感受系统、情绪、行动等

状态池 SP 存放运行时“当前活跃的认知对象”。对象可以是：
- 结构（ST）或结构相关对象（如结构组/事件结构）
- 刺激元/属性刺激元在一定阶段的投影
- 行为候选与内部感受等（依工程阶段而定）

### 3.2 双能量：实能量 ER 与虚能量 EV
系统对每个可竞争对象维护两种能量：
- **实能量 ER**：来自直接证据、当前输入或被验证的成分；
- **虚能量 EV**：来自预测、推演、联想、记忆传播等。

二者共同决定对象的“存在感”和“被处理优先级”，并通过差值产生认知压。

### 3.3 认知压 CP
定义：`CP = |ER - EV|`。  
直觉：当一个对象“实与虚不平衡”时，系统更倾向于把注意力资源投入到它身上进行解释/中和。  
重要约束：并不把 CP 简化为惩罚。CP 更像“需要解释的张力”，它可以驱动注意力聚焦、记忆检索、内源刺激化等，但不等价于危险。

### 3.4 结构（Structure, ST）与残差信息（Residual）
结构 ST 是长期记忆层（HDB）中可被局部打开、可被复用的认知单元。每个 ST 维护一个局部数据库，包含：
- 与该结构共同出现过的残差信息（可理解为“在出现该结构时，剩余部分通常是什么”）
- 残差信息的权重（共现/时序统计意义上的可能性）
- 可能包含时序组信息（哪些残差在前/后，或属于同一时序组）

残差信息可以分为两类：
- **共现残差**：表示“同时出现”的关联（顺序弱或无）
- **时序残差**：表示“先后/角色/语法位”的关联（顺序敏感）

### 3.5 结构组（Structure Group, SG）与事件结构（Event Structure, ES）
在既有体系里，SG 常被用作结构级阶段的组合结果，更多表达关系而非对象。  
本文档提出：让“事件认知”成为真正的结构对象 ES：
- ES 由多个结构（或更小对象）按一定顺序/关系组成；
- ES 有自己的 ER/EV、CP、半衰期与淘汰机制；
- ES 拥有可索引的数据库入口（直接或间接），并能产生残差信息；
- ES 进入状态池，与 ST 同权参与注意力竞争。

### 3.6 内源性刺激与外源性刺激
- 外源性刺激：来自环境输入（用户消息、系统事件、API 返回、操作反馈等）。
- 内源性刺激：来自系统内部的回忆、残差、预测、事件组织后的“内心材料”等。

在拟人陪伴机器人场景下，内源性刺激在长期运行中理应占有较大比重：它承担持续思考、回顾、推演与叙事维持的角色。外源刺激是触发与校准，不应长期压倒一切。

## 4. 结构级查存一体与刺激级查存一体的功能边界：重复与不可替代性

### 4.1 看似重复的原因
若历史上发生过某事，则该事的外源刺激与当时代表状态池现状的内源刺激都会进入刺激级查存一体并写入记忆。于是，“状态池状态信息”似乎已经被刺激级记忆覆盖；当现状与历史相似时，刺激级查存一体可能直接命中最大匹配结构，从而把经验带回来。

这使得一个问题变得尖锐：结构级查存一体是否只是重复劳动？

### 4.2 刺激级查存一体天然做不到或不擅长的部分
刺激级查存一体的主要对象是“刺激包”，核心操作是：
- 在局部库范围内做贪婪链式匹配；
- 基于最大共同部分切割形成共同结构并写回；
- 把残差写入局部库；
- 以刺激包为粒度表达时序组与共现关系。

它确实能形成结构化记忆，但它天然不擅长以下目标：
- **运行期事件对象的稳定存在**：刺激级的产物多是“解释结构”，并不必然作为一个长期可竞争的整体对象留在状态池里。
- **把“状态组合 -> 操作/回顾倾向”表达为局部可打开的策略线索**：刺激级命中的是输入刺激形态；而“状态池现状”是一种更高阶的组合，它需要在运行期被组织成可以竞争的实体，才能直接影响注意力与行动。
- **资源纪律与局部化入口控制**：刺激级若承担过多“事件组织”的职责，容易退化为对大量内源刺激的重复处理，导致候选爆炸与耗时放大。

因此，结构级查存一体在理论上仍有存在空间：它试图把状态池（CAM）作为输入，直接把“当前状态组合的经验”带回。然而既有实现的结构级路径已经出现明显过热与膨胀，需要一种新的替代机制来承担“事件组织”的主要职责，并把结构级路径降为可选对照。

### 4.3 CS 的立场：不是否定结构级，而是把“事件组织”前移到状态池维护后
CS 的关键假设是：
- 事件组织不应依赖高成本的结构级多轮检索；
- 事件组织可以利用 HDB 的局部残差（每个结构自带的局部知识），在状态池中做近似 O(local) 的拼接；
- 事件组织后的 ES 进入注意力竞争，使“经验应用”以更自然的整体对象形式发生；
- 结构级查存一体保留为开关与对照，未来可退化为“低频深检索”或“闲时巩固”阶段，而不是每 tick 的主干成本中心。

## 5. 设计哲学与工程约束

### 5.1 尽量少硬规则：让系统涌现而不是手写禁令
本文档贯彻以下原则：
- **连续化评分优先于二元 gate**：能用连续变量表达的，不用 if/else 直接否决；
- **能量与竞争优先于手工上限**：长度、复杂度、叙事唯一性等倾向，尽量由能量吸收、衰减、软上限与竞争自然产生；
- **弱命中优先于“不允许匹配”**：顺序证据不足时，允许低置信度匹配，保留中间态；
- **可观测优先于“相信它会对”**：任何关键机制都必须有审计字段与长期图表，否则调参将退化为玄学。

### 5.2 资源纪律：预算必须能从系统内部变量推导
CS 的资源纪律来自三类可审计约束：
- **能量守恒近似**：ES 的生成/扩展吸收组分能量，防止“凭空增加复杂结构”；
- **半衰期与淘汰**：长期运行中，低能量组分自然掉队，事件自然缩短或消失；
- **容量软上限按“总长度”计费**：状态池的软上限不只数对象个数，也可数对象内部长度（组分数或展开刺激元数），避免用一个超长 ES 绕开容量约束。

### 5.3 认知压的双重作用：差异大促碎片化，差异小促整体化
在 AP 的双能量语义下，认知压 `CP=|ER-EV|` 不应被简化为惩罚，它更像“需要解释的张力”。  
认知拼接希望 CP 的变化自然诱发两种相反倾向：
- 当 CP 变大：系统更倾向把注意力拉向局部差异源，促使碎片化认知、解释与重组。
- 当 CP 变小：系统更倾向保留整体事件结构，以更少对象承载当前态势，节省资源。

这使得整体化并非单向增长，而是与“差异是否被中和验证”紧密耦合：越被验证、越平衡，越整体；越不平衡、越需要拆解。

### 5.4 允许“统一叙事确信”与“偶尔犯错”，并依靠长期自校准回收
拟人系统不追求逻辑绝对正确，而追求拟人过程。人类天然偏好简洁统一的解释，甚至把想象当作见过。  
认知拼接允许事件结构短期形成较强确信（甚至带一定 ER），并进入状态池竞争与影响行动倾向；同时依靠统一的长期自校准回收错误叙事：
- 若事件主要由实证据支撑，边权与事件结构会逐步稳定。
- 若事件主要由虚证据支撑且反复不验，虚证据磨损会逐步回收其权重，并通过 CP 拉高促使注意力回到碎片进行重解释。

### 5.5 空间换时间：运行时轻、存储侧重、闲时巩固
事件结构的组合空间很大，但运行时必须保持局部化与低算力根基。因此本设计坚持：
- 运行时尽量不做重型物理融合（parents+delta 懒融合、top-k 合成缓存）。
- 闲时或数据集结束时触发巩固，把父链压平、去重压缩、释放内存。
- 该巩固必须可被前端显式触发并可审计（否则无法验收）。

### 5.6 可回滚与 A/B：结构级路径保留，但默认关闭
为了验收严格与可控，工程上必须提供开关：
- `enable_structure_level_retrieval_storage`：结构级查存一体流程开关，默认关闭；
- `enable_cognitive_stitching`：CS 模块开关，默认开启。

两者可组合：
- 仅结构级：回到旧机制对照；
- 仅 CS：主推新机制；
- 两者都开：用于混合或过渡（需谨慎，避免双重写回导致膨胀）；
- 两者都关：用于最小闭环诊断。

## 6. 认知拼接模块总览（CS 的位置、输入输出、开关策略、模块契约与边界）

### 6.1 CS 在 Tick 主链中的位置
CS 的插入点固定为：
1. 状态池维护完成  
2. **CS：在状态池快照上构造/更新事件结构 ES，并把 ES 放回状态池**  
3. 注意力筛选（AF）：ES 与普通结构共同竞争进入 CAM  
4. 后续结构级/刺激级查存一体（由开关决定）

选择这一位置的理由：
- 事件认知应影响注意力：把“注意到某件事”提前到 CAM 形成之前；
- 事件认知应利用状态池现状：CS 需要看到本 tick 的活跃结构分布与能量图景；
- 事件认知不应依赖结构级多轮检索：CS 应是轻量局部过程。

### 6.2 输入与输出
输入：
- 状态池快照：包含当前活跃结构（ST）与已有 ES（若存在）的集合及其 ER/EV/CP；
- HDB 局部访问能力：能通过结构 id 或 db pointer 打开局部残差表的 top-k；
- 配置：拼接候选宽度、弱命中参数、能量吸收比例、长度治理策略等。

输出：
- 新建或更新后的 ES 列表（作为结构对象写回状态池）；
- 拼接审计报告：本 tick 尝试了哪些候选、为何胜出、消耗了哪些能量、剪枝原因等；
- 统计指标：供前端与 APT 使用。

### 6.3 与结构级查存一体的关系
CS 并不要求删除结构级查存一体；推荐默认策略是：
- 结构级查存一体关闭（避免过热与重复）；
- CS 开启，作为事件组织的主机制；
- 刺激级查存一体保持开启（保留原有学习与解释能力）；
- 未来在“闲时/空 tick”阶段可选择性启用结构级深检索作为巩固，而非每 tick 常驻。

### 6.4 模块契约（Module Contract）
CS 是运行态的“事件组织层”模块，其契约可以用一句话概括：
在不进行全局扫描、不引入大量硬规则的前提下，把状态池中的离散结构对象组织为少量可竞争的事件结构（ES），并以能量账本约束其生成、扩展与收缩，使“整体化 vs 碎片化”的切换由认知压与证据结构自然驱动。

CS 在每 tick 至少应完成以下职责：
- 从状态池快照中选取有限数量的拼接锚点（anchors），并按局部残差 top-k 产生候选拼接动作。
- 对候选动作进行连续评分与竞争选择（不使用二元 gate 作为主机制）。
- 执行能量账本：从参与组分扣减 ER/EV，并把扣减能量汇聚到 ES（近似守恒，允许轻微损耗作为稳定器）。
- 更新 ES 的 `composition_profile`（组分能量比例画像）与必要的审计字段。
- 执行事件收缩：当组分分摊能量低于阈值时剔除组分，必要时触发“最大可匹配前缀”写回残差，保证事件结构长期可演化且不无界膨胀。
- 输出可审计报告与 metrics 字段，供前端观测与 APT 调参使用。

### 6.5 模块边界（Non-goals）
CS 不承担、也不应“偷做”的职责包括：
- 不替代刺激级查存一体的“最大共同部分切割式结构学习”。CS 的学习粒度是结构引用（ST/ES），不是 SA 级别的切割学习。
- 不直接执行感应赋能（传播与推演）。CS 只组织对象与记账；传播仍由感应赋能阶段统一执行。
- 不直接生成外显行动。CS 只能通过影响注意力候选与能量图景间接影响驱动力与行动竞争。
- 不把“事件正确性”当作硬逻辑约束。事件允许短期确信并犯错，长期依靠实证据增强与虚证据磨损自校准回收。

CS 的概念级输入输出接口建议稳定为：
- 输入：`state_pool_snapshot`、`hdb_access`、`tick_id`/`trace_id`、`cs_config`。
- 输出：`state_pool_updates`（新增/更新/裁剪 ES、能量扣减与转移）、`cs_report`（brief/detail/trace 三层）、`metrics_fields`。

## 7. 事件结构（Event Structure, ES）：让结构组成为“真正的结构”（双视图、规范签名、composition profile、总容量口径）

### 7.1 ES 的最小数据结构
ES 作为结构对象进入状态池，至少应包含：
- `es_id`：事件结构 id（全局唯一）
- `components`：组分列表（元素是结构 id；允许组分本身是 ES，形成层级，但需谨慎）
- `order_model`：顺序模型（例如时序组/有向边/角色位等的轻量表示）
- `composition_profile`：组分能量比例信息（记录 ES 能量在组分间的分配比例）
- `energy`：ES 自身的 `(ER, EV)` 与衍生 CP
- `hdb_pointer`：指向 ES 的局部数据库入口（可能是直接指针，也可能是懒融合的 overlay 描述）
- `provenance`：来源（由哪些拼接动作形成，吸收能量多少，父 ES/父 ST 列表等）
- `audit`：可选审计字段（最近一次拼接得分、弱命中系数、使用的上下文长度等）

### 7.2 ES 的“完备性”定义
为了避免出现“半结构、不可索引”的技术债，ES 的完备性定义如下：
- ES 必须能通过 `es_id` 在 O(1) 时间打开其局部数据库入口（直接或间接）。
- 若 ES 对应数据库不存在，则必须有可追溯的“构建方式”：至少能由父结构数据库懒融合得到可用视图，并能在闲时巩固后固化为独立数据库。

因此 ES 的数据库可以分两类：
- **显式数据库（materialized DB）**：有独立存储与索引；
- **懒融合数据库（overlay DB）**：由 `(parents + delta)` 描述，运行时可打开视图，闲时可固化。

### 7.3 ES 的语义：事件是“整体对象”，但允许内部碎裂
ES 被当作整体对象进入注意力与展示，但并不要求内部组分永远不变。内部碎裂来自能量治理：
- ES 在状态池中整体衰减；
- 通过 `composition_profile` 给组分分配“组分实时能量”；
- 当某组分实时能量低于淘汰阈值时，该组分可从 ES 中移除，形成更短的新 ES；
- 新 ES 若在 HDB 中存在对应数据库，则合并旧信息；若只匹配到最大前缀，则把剩余部分作为残差写回（见后续章节）。

这种“整体存在但可碎裂”的特性，使事件既能产生拟人叙事的稳定感，又能在证据变化时退回碎片化解释的模式。

### 7.4 ES 的双视图：时序视图与共现视图（为弱命中预留）
为支持“顺序敏感连续谱”和弱命中（第 9 节），ES 建议同时维护两套可匹配视图：
- 时序视图（sequence view）：保留组分的先后顺序与时序组边界，用于顺序证据强时的强命中与高置信拼接。
- 共现视图（cooccurrence view）：将一部分组分折叠为无序信息组（组内可交换），用于顺序证据弱时的弱命中与低幅度验证。

双视图的意义在于：
- 不把“共现=无效”写成硬门槛，而是让缺少时序证据的内源性刺激仍可对事件结构产生小幅影响。
- 不把“共现=顺序”混为一谈：共现视图命中强度上限更低、权重更新更谨慎，避免把纯共现误当作动作叙事（例如“暴打柠檬”的动词短语 vs 名词短语差异）。

### 7.5 规范签名（Canonical Signature）与唯一性（Pointer Index）
事件结构若进入状态池并可长期演化，必须解决“同一事件结构如何被认出是同一对象”的身份一致性问题。这属于基础设施，不属于硬规则限制行为。

定义 ES 的规范签名 `sig(ES)`，用于：
- 在创建 ES 前做复用查找（避免重复建库）。
- 在 HDB 中维护 `pointer_index: sig -> (es_id, hdb_pointer)` 的 O(1) 入口。

签名建议至少包含以下信息：
- 组分结构 id 的序列（或稳定内容签名）。
- 时序信息组边界（用于区分 `[A,B,C]` 与 `[{A,B},C]` 这类差异）。

为了支持弱命中与“中间态演变”，建议同时维护两类签名：
- `sig_seq(ES)`：严格保序签名（用于强命中、强复用）。
- `sig_set(ES)`：折叠/排序后的共现签名（用于弱命中候选定位与“近似复用”）。

注意：`sig_set` 只能用于候选定位与弱证据，不应直接替代 `sig_seq` 的唯一性判定，否则会把“相似”误当“相同”并导致事件硬化。

### 7.6 ES 组分能量画像（composition profile）与组分账本
`composition_profile` 是 ES 的核心结构性信息之一，它承担三个角色：
- 长度治理：把 ES 的总能量分摊到组分，决定哪些组分会因能量过低被剔除。
- 匹配评分：用于衡量“当前能量图景”与“历史该事件图景”的相似度（第 8 节与第 16 节）。
- 中和联动：为“事件内部组分互补中和”提供可审计的能量分配依据（第 13 节）。

工程上建议区分两层：
- 比例画像（profile）：记录比例 `p_i`（以及可选的 ER/EV 分别比例）。
- 组分账本（ledger）：记录组分内部分配的 `(ER_i, EV_i)`，并满足 `Σ ER_i = ER_es`、`Σ EV_i = EV_es`。账本用于中和与一致性审计，不必每 tick 全量更新，但在发生拼接/裁剪/中和时必须可更新。

### 7.7 总容量口径：对象数、组分数、刺激元数（可切换，双口径必记录）
为了避免“用一个超长 ES 绕开状态池容量约束”，本设计要求状态池容量治理支持“总容量”口径，并且口径可配置：
- 口径 A：按对象数计费（兼容旧模式）。
- 口径 B：按 ES 组分数计费（默认建议先采用，稳定且成本低）。
- 口径 C：按展开后的刺激元数估计计费（更贴近语义复杂度，但成本更高）。

无论采用哪种口径治理，长期指标必须同时记录两套长度统计：
- `event_len_components_*`：ES 组分数口径。
- `event_len_stimulus_units_est_*`：刺激元估计口径。

## 8. 拼接算法（从局部残差到事件结构的生成与演化）

本节描述 CS 在一个 tick 内部的基本工作流。核心目标是：在局部信息与有限预算下，生成少量高价值 ES，而不是遍历所有组合。

### 8.1 锚点选择（Anchors）：在局部预算下确定可拼接对象
CS 的第一步不是“到处找可拼接对”，而是确定少量锚点对象（anchors）。锚点集合定义了本 tick 的拼接搜索空间上界，从而保证成本与当前活跃状态相关，而不是与全库规模相关。

锚点来源建议包含两类：
- 高能结构：状态池中 `ER+EV` 较高的 ST/ES，代表“当前真正活跃的内容”。
- 高压结构：状态池中 `CP=|ER-EV|` 较高的 ST/ES，代表“当前需要解释/中和的张力源”。

锚点选择必须受预算约束（可由配置与 APT 调参）：
- 每 tick 仅选取前 `K_anchor` 个锚点。
- 或按“预算扣减式”选择：每选一个锚点扣除少量 CS 预算，预算耗尽即停止。

锚点选择的结果应进入观测字段（例如 `cs_anchor_count`、`cs_anchor_energy_sum`），否则无法判断“候选空间为何过宽”。

### 8.2 拼接的原子：边（Edge）与上下文（Context）
拼接可以被抽象为在结构序列上添加边：
- `A -> B` 表示在某种时序/角色语义下，A 后面倾向于出现 B；
- 或在共现语义下，A 与 B 可组成更稳定的整体。

但 CS 的关键增强是“上下文边”：
- 不仅允许 `D -> E` 作为 `[A+B+C+D] + E` 的依据；
- 还允许 `suffix([A+B+C+D]) = [C+D] -> E` 或 `[B+C+D] -> E` 等更长上下文提供更强证据；
- 同理，前缀扩展允许 `X -> prefix([A+B+...])` 的多长度依据。

上下文越长，匹配越强；但越长的上下文也越稀疏，因此需要回退机制。

### 8.3 候选生成：局部残差 top-k
对状态池中的每个候选节点（ST 或 ES），通过 HDB 打开其局部残差表，取 top-k 作为候选边：
- 共现残差候选：`A ~ B`
- 时序残差候选：`A -> B`，并携带时序组信息
- 允许残差项本身是复合结构（如 `A+B` 作为一个残差结构 id）

为了保证局部化，候选生成必须满足：
- 不做全库扫描；
- 不在运行期打开与当前活跃对象无关的数据库；
- top-k 的 k 可配置，并可被 APT 调参。

候选边（residual link）为了支持“顺序/共现双视图”“实证据增权、虚证据磨损”“疲劳与新鲜度”等统一语义，建议至少携带以下字段（字段名可按工程规范调整）：
- `target_id`
- `edge_mode`：sequence / cooccurrence（或连续化倾向值）
- `base_weight`
- `recent_gain`
- `fatigue`
- `evidence_er_sum`：实证据累计（增权来源）
- `evidence_ev_sum`：虚证据累计（磨损来源）
- `last_hit_tick`
- `sequence_group_hint`：时序对齐提示（可选）

事件到事件的拼接不应被视为独立特例，而应被统一理解为“上下文边”的一般情形。设左事件 `L=[A,B,C,D]`、右事件 `R=[E,F,G,H]`，则以下证据都属于可比较的桥接候选：
- `D -> E`
- `[C,D] -> E`
- `D -> [E,F]`
- `[C,D] -> [E,F]`

若顺序完全一致，则命中更强；若顺序部分错位，但关键锚点仍可对齐，则只以弱命中形式参与竞争。这样，“事件组与事件组拼接”就不需要额外发明一套例外规则，而是被纳入统一的 suffix/prefix backoff 与弱命中体系。

为了避免把“raw weight 大”误当作“桥接强”，候选边的有效强度应优先采用“源库内占比 + 出流总量”的联合量，而不仅是绝对值。一个可审计的示例形式是：
`p_rel = (w_target + α_smooth) / (Σ_j w_j + α_smooth * N_out)`
`v_out = Σ_j w_j / (Σ_j w_j + κ_vol)`
`w_ratio_eff = p_rel * v_out`

其中：
- `p_rel` 表示该目标边在当前源上下文所有可走边中的相对支配度；
- `v_out` 表示该源上下文本身是否已经积累了足够出流证据，避免“小样本高比例”的伪强边；
- `w_ratio_eff` 作为 `W_term` 的基础输入，再叠加新鲜度、疲劳与 ER/EV 调制。

此外，建议对“源上下文 -> 目标上下文”这一同类桥接路径维护短时抽能疲劳，而不是用硬编码阻止重复拼接。可定义：
`drain_fatigue(sig) = 1 / (1 + λ_drain * n_recent(sig))`

其中 `sig` 是桥接签名（例如 `suffix_k(L) -> prefix_q(R)`），`n_recent(sig)` 是短窗口内已发生的相同抽能次数。该因子用于缩放评分与吸收比例，使重复抽能越来越弱，但永不强制归零。

### 8.4 竞争评分：用连续变量替代硬 gate
对每个“拼接动作候选”，计算一个综合评分 `Score`，并通过竞争选出赢家。评分建议包含以下因子（均为连续）：
- `E_term`：能量项（候选对象的 ER/EV 与 CP）
- `W_term`：残差权重项（优先使用“源库内相对占比 + 出流总量”的联合量，而不仅是 raw weight，且需考虑 ER/EV 对权重的增强/削弱）
- `C_term`：上下文长度项（使用的 suffix/prefix 长度越长越高）
- `M_term`：弱命中匹配项（顺序证据越一致越高）
- `R_term`：同一路径短时重复抽能疲劳项（重复越多越弱，但不归零）
- `G_term`：带孔子序列的 gap 惩罚项（允许跳过但扣分）
- `D_term`：多义/冲突惩罚（例如该候选会与已有强事件冲突时扣分，扣分应连续化）

一个可审计的示例形式：
```
Score = a*log(1+E_total) + b*W_ratio_eff + c*log(1+context_len)
        + d*match_sigma + e*drain_fatigue
        - f*gap_penalty - g*conflict_penalty
```
其中 `W_ratio_eff` 是按相对占比与出流总量联合标定、再经 ER/EV 调制后的有效权重（见 12.4）。

赢家选择不建议硬编码为“只能有一个”，更推荐“软互斥”：
- 候选按 `Score` 排序逐个尝试执行。
- 一旦某个候选执行完成，参与组分能量被吸收扣减，其它候选的 `E_term` 会自然下降，从而降低再次胜出的概率。
- 再叠加“每 tick 最大拼接次数”“每组分每 tick 最大可吸收能量”等预算语义，即可形成稳定的软互斥，而无需把“只能属于一个事件”写成硬规则。

### 8.5 拼接动作的类型（Create / Extend / Prepend / Merge / Prune）
为了让 CS 可直接落地并可审计，本设计把拼接动作限定为 5 类原子操作：
1. **Create**：由两个或多个结构创建最小事件结构（例如 `[A,B]`）。
2. **Extend**：在事件右端追加一个结构（例如 `[A,B,C] + D -> [A,B,C,D]`）。
3. **Prepend**：在事件左端前置一个结构（例如 `X + [A,B,C] -> [X,A,B,C]`）。
4. **Merge**：合并两个事件结构（通常允许存在重叠锚点，如 `[C,A,B]` 与 `[B,D]` 合并为 `[C,A,B,D]`）。
5. **Prune**：按能量阈值剔除低能组分，使事件收缩为更短事件（例如 `[A,B,C,D] -> [A,B,D]`）。

动作 2/3 应优先使用上下文回退（后续章节详细定义）。

其中，**Merge** 应被明确扩展为“事件组到事件组”的桥接拼接，而不仅是短事件的局部并接。若左事件 `L=[A,B,C,D]`、右事件 `R=[E,F,G,H]`，则桥接依据可以来自 `suffix_k(L)` 与 `prefix_q(R)` 的任意组合。桥接时：
- 实际参与对齐的对象越多，形成整体的可信度越高，但增益应采用次线性饱和形式，避免长事件线性垄断。
- 上下文越长、顺序越一致、目标边在源库中的相对占比越高，则吸收能量越多。
- 若仅出现弱命中或部分错序，则允许形成弱整体，但吸能更少，原有子事件仍保持主导。

端点扩展原则（默认）：Extend/Prepend 默认仅允许在两端增长，不在中间插入新组分，以避免组合爆炸并增强“句子骨架”的可读性。  
该原则不排斥“带孔子序列”与“上下文回退”（第 10 节），后者允许用更长上下文作为证据，但事件本体仍只在端点增长。

### 8.6 能量吸收：生成与扩展必须付出代价
无论哪种拼接动作，ES 的生成/扩展都要吸收组分能量。基本原则：
- 不能凭空制造一个高能量的整体；
- 事件压缩会让碎片变弱（更易淘汰），以换取整体更稳；
- 吸收比例不固定，而是由候选残差权重与匹配度决定（越确信、越一致，吸收越多）。

具体计算方式见第 11 节。

### 8.7 组分淘汰与事件收缩：用能量而不是硬长度上限
ES 在长期运行中可能变长，但“无限生长”不是默认结局。原因在于：
- 能量整体衰减；
- 状态池软上限会加速衰减；
- 组分能量按比例分配，低于阈值的组分会被剔除；
- 剔除触发事件收缩，并可能引发 HDB 写回（把失配部分作为残差）。

长度治理的形式化分析见第 11 节。

## 9. 弱命中（Weak Hit）与顺序证据（顺序敏感连续谱、os(X)、σ_seq/σ_set/γ 分解）

### 9.1 动机：内源刺激的顺序弱，但仍应能命中顺序结构
在拟人陪伴机器人中，内源性刺激经常是“共现型材料”：
- 想到几个概念、几个对象、几个情绪信号；
- 并不总带明确时序或语法角色。

若把“顺序敏感”做成严格 gate，则内源刺激几乎无法命中时序结构，导致：
- 内源刺激的记忆利用效率下降；
- 事件组织被迫依赖外源刺激或硬规则；
- 系统失去“动态演化与中间态”的美学。

因此需要弱命中：允许顺序证据不足时仍匹配，但匹配强度低。

### 9.2 顺序证据 γ：来自输入与目标的双因素
定义一个连续系数 `γ ∈ [0,1]` 表示“本次匹配中顺序约束的可信度/重要性”。它可以由两部分决定：
- 输入刺激的顺序证据 `γ_in`：外源刺激与残响往往带更强顺序；内源刺激更弱；
- 目标结构的顺序敏感度 `γ_tgt`：某些结构天然顺序敏感（动词短语、操作序列、角色位）；某些结构更共现（主题、属性集合）。

一个可解释的组合：
`γ = clamp(γ_in * γ_tgt, 0, 1)`

### 9.3 匹配强度 σ：集合匹配与序列匹配的凸组合
设：
- `σ_set`：集合/共现层面的匹配（忽略顺序）
- `σ_seq`：顺序层面的匹配（考虑时序组/边/角色位）

定义弱命中匹配强度：
`σ = γ*σ_seq + (1-γ)*σ_set`

这使得：
- 外源性、顺序证据强、目标顺序敏感：σ 主要由 σ_seq 决定（强命中）
- 内源性、顺序证据弱：σ 主要由 σ_set 决定（弱命中但不为零）

### 9.4 示例：顺序敏感与共现的混合
例 1：“暴打柠檬”  
- 若结构表示一个动作序列（动词打 + 宾语），则顺序敏感度高；  
- 若输入只是内源性想到“暴打”“柠檬”两个词，共现匹配可能较高，但顺序证据低，因此 σ 处于弱命中区间；  
- 若外源输入出现明确句式或操作事件（例如“执行：暴打柠檬”），顺序证据升高，σ 上升为强命中。

例 2：“紫苹果”  
若系统从“紫色”“苹果”共现拼接得到 ES：  
- 初期可能以 EV 为主，事件整体存在但把握感低；  
- 后续外源刺激确证“苹果”出现，顺序证据并非关键，σ_set 足以支撑匹配；中和过程会降低组分 CP（见第 13 节），从而提升事件把握感（第 14 节）。

### 9.5 顺序敏感度 os(X)：从二元属性改为连续谱
在真实认知中，“共现关系”与“时序关系”并非互斥。为了兼容外源残响的局部时序骨架与内源刺激的共现团表达，本设计把“顺序敏感”从二元 gate 改写为连续谱参数：

`os(X) ∈ [0,1]`

直观解释：
- `os≈0`：更像共现团或属性集合，顺序不重要。
- `os≈1`：更像动作/事件链或角色位结构，顺序重要。
- 中间值：部分顺序重要、部分可弱对齐。

`os(X)` 的来源不要求硬编码规则。推荐策略是：
- 冷启动给一个保守默认值（例如 ST 的 `os` 中等偏低，ES 的 `os` 中等偏高）。
- 长期由命中统计涌现更新：当结构的高质量命中主要来自时序对齐时 `os` 上升；当主要来自共现弱对齐时 `os` 下降。

### 9.6 γ 的构造：顺序证据来自“输入”与“目标”的共同作用
在第 9.2 节中，γ 被拆为输入与目标两部分。本节给出一个更便于工程落地且可审计的表达：

- 定义刺激的时序证据强度 `T(I) ∈ [0,1]`：外源残响更高、纯内源共现团更低。
- 目标结构的顺序敏感度使用 `os(X)`。

则：
`γ = clamp(T(I) * os(X), 0, 1)`

该定义的意义是：
- 输入没提供顺序证据时（T 低），即便目标顺序敏感，也只能弱命中。
- 输入顺序证据强时（T 高），目标越顺序敏感越倾向强命中。

### 9.7 弱命中如何作用于能量转移与权重更新（幅度缩放，而非离散裁决）
弱命中机制的关键不是“允许/禁止”，而是“幅度缩放”。建议把 σ 直接用于缩放能量与权重更新的幅度：

- 能量转移（概念表达）：
`ΔE_transfer = σ * E_evidence_subset`

- 权重更新（与“实证据增权、虚证据磨损”一致）：
`Δw += k_r * σ * E_r_evidence`
`Δw -= k_v * σ * E_v_evidence`

其中 `k_r`、`k_v` 为可配置系数，并应纳入 APT 可调参空间。

该设计保证：
- 弱命中不会凭空把事件变强，它只是允许在证据不足时发生小幅影响。
- 若事件长期被现实验证，σ 会随 T(I) 与 os(X) 的统计演化逐步提高，事件结构自然变“更像逻辑”。
- 若事件主要靠虚能量维持，磨损项会逐步回收其权重，避免推演泡沫长期固化。

## 10. 上下文回退拼接（Suffix/Prefix Backoff）与带孔子序列（Anchored Subsequence）

### 10.1 需求：拼接依据不应局限于端点
扩展 `[A+B+C+D]` 与 `E` 的拼接，不能只允许 “D 的残差包含 E”。  
合理的依据应包括：
- `suffix2 = [C+D]` 的数据库中包含 `E`
- `suffix3 = [B+C+D]` 的数据库中包含 `E`
- `suffix4 = [A+B+C+D]` 的数据库中包含 `E`

同理，前缀扩展 `X + [A+B+C+D]` 也允许：
- `X` 的残差包含 `A`
- 或包含 `A+B`
- 或包含 `A+B+C`
- 或包含与前缀一致的更长片段。

其核心是：**上下文越长越好，但必须允许回退**，并且回退是“连续削弱”而不是“完全不允许”。

同一原则也适用于事件组到事件组的桥接。若左事件为 `[A+B+C+D]`，右事件为 `[E+F+G+H]`，则：
- `D` 的数据库内包含 `E`，可作为最弱桥接依据；
- `D` 的数据库内包含 `E+F`，说明右事件前缀有更强局部支持；
- `[C+D]` 的数据库内包含 `E` 或 `E+F`，说明左侧后缀上下文对右侧前缀具有更高定向性；
- 若顺序存在偏差，但关键锚点仍可对齐，则仍允许弱命中，只是评分、吸能与后续权重增强更弱。

因此，“事件组也可以进行拼接”并不是新增例外，而是上下文回退拼接在事件层上的直接推广。

### 10.2 Suffix/Prefix Backoff 的形式化
定义 ES 的组分序列为 `S = [s1, s2, ..., sn]`。

后缀上下文集合：
`SuffixCtx(S) = { [sn], [s(n-1), sn], ..., [s1,...,sn] }`

前缀上下文集合：
`PrefixCtx(S) = { [s1], [s1,s2], ..., [s1,...,sn] }`

对于要追加的候选 `E`，允许任意 `ctx ∈ SuffixCtx(S)` 提供证据：`ctx -> E`。
评分中加入上下文长度项，例如：
`C_term = log(1 + len(ctx))`
并为回退引入惩罚，使短 ctx 仍可用但更弱。

### 10.3 带孔子序列（Anchored Subsequence）：顺序一致但允许跳过
在某些情况下，拼接依据可能来自非连续片段，例如：
- `X` 的数据库包含 `[A+B+D]`，而当前 ES 前缀是 `[A+B+C+D]`。  
这表示“关键锚点序列一致”，但中间有一个孔（跳过了 C）。

这类依据可以被视为“带孔子序列”：
- 允许作为弱证据；
- 需要对 gap 数量与位置施加连续惩罚；
- 仍然要求相对顺序一致（不允许 `[B+A]` 匹配 `[A+B]`）。

形式化地，给定目标序列 `T` 与锚点序列 `U`：
- 若 `U` 是 `T` 的一个有序子序列，则可匹配；
- gap 惩罚可取：
`gap_penalty = λ * (|T| - |U|)` 或更复杂的加权 gap（越靠近端点孔惩罚越大）。

### 10.4 为什么允许带孔：拟人直觉与资源效率
人类推理常常使用“关键线索链”而不是完整连续匹配。允许带孔的价值在于：
- 在稀疏数据下仍能形成合理事件；
- 形成“直觉式但不严格”的推演路径；
- 通过能量账本控制其影响范围：证据弱则吸收能量少、权重低、把握感低。

### 10.5 回退式上下文搜索算法（Backoff Search，右端扩展示例）
回退式拼接的落地关键在于：不枚举所有可能上下文，而是在预算内生成一个“从长到短”的上下文候选序列，并沿着局部索引链逐级回退。

以右端扩展为例，给出最小可落地流程：
1. 生成上下文集合  
按预算优先生成连续后缀上下文：`[sn]`、`[s(n-1),sn]`、...、`[s(n-k+1),...,sn]`（k 不超过 `K_max`）。  
当连续上下文找不到有效边且仍有预算时，再补少量“右端锚定”的带孔子序列上下文，并记录 `gap_count`。
2. 上下文结构定位（O(1)）  
对每个上下文 `ctx` 计算 `sig_seq(ctx)`，查 `pointer_index` 得到其结构 id 与数据库入口；若不存在则回退到更短上下文（或直接跳过该 ctx）。
3. 打开上下文数据库并寻找后继边  
打开 `ctx` 的局部残差 top-k，生成后继候选 `ctx -> E`（其中 E 必须在状态池活跃或满足最小能量阈值，避免凭空扩展）。
4. 连续评分与竞争  
对每个候选动作 `Extend(ES, E, ctx)` 计算 `Score`，其中：
- 上下文长度越长，`C_term` 越大；
- `gap_count` 越大，`G_term` 惩罚越强；
- 弱命中 σ 用于缩放能量转移与权重更新（第 9.7 节）。
5. 执行胜出动作并记账  
按 `Score` 从高到低尝试执行，执行后能量被吸收扣减，形成软互斥。执行结果与停止原因必须写入审计字段（例如 stop_by_budget / stop_by_no_candidate）。

左端前置（Prepend）对称，只需把连续上下文替换为前缀上下文，并使用“左端锚定”的带孔子序列上下文。

### 10.6 “包含/匹配”的语义：顺序敏感、角色敏感，但允许弱命中
上下文是否“包含”某个片段，不能仅靠共现集合判断。默认语义应以时序骨架为主：
- 对顺序敏感结构：顺序一致是强证据，顺序不一致只能作为弱命中或不命中。
- 对共现结构：允许以共现证据产生弱命中，但其 `σ_set` 上限与权重更新幅度应更低，避免把纯共现误写成强时序叙事。

该语义与第 9 节的连续谱弱命中一致：不是“能/不能”，而是“证据强弱与更新幅度”。

## 11. 能量账本：近似守恒、吸收比例、长度治理与“不会无限生长”的条件

### 11.1 近似守恒的必要性
若 ES 生成不消耗能量，则系统可以无成本地生成大量组合事件，导致：
- 状态池对象数量暴涨或被超长 ES 隐性绕开；
- 内源刺激爆炸；
- 事件竞争失真，形成不可控的“叙事泡沫”。

因此规定：ES 的生成与扩展必须吸收组分能量，并将其从组分对象中扣除。该吸收是“压缩代价”。

### 11.2 吸收比例 α 的连续化定义
吸收比例不应是定值。建议让 α 由以下因素共同决定：
- 残差权重归一化比例（代表统计共现可能性）
- 匹配强度 σ（弱命中/强命中）
- 上下文长度（更长上下文更可信）
- 组分 CP（高 CP 的组分不宜过度吸收，以免掩盖需要解释的张力）

一个示例：
```
α = clamp( α0
           + k_w * W_norm
           + k_m * σ
           + k_c * log(1+context_len)/log(1+L_max)
           - k_p * CP_norm,
           α_min, α_max )
```
其中 `W_norm` 是候选残差权重占该数据库 top-k 权重和的归一化比例。

在工程落地上，也可把 α 拆成更可审计的乘法分解形式（两者等价，区别在于调参手感）：
`α = clamp(α_global * f_edge * f_len * f_budget * f_match * f_cp * f_part * f_drain, α_min, α_max)`
其中：
- `f_edge` 来自边在源库中的归一化占比与出流总量联合量（越“定向且有量”的后继越大）。
- `f_len` 来自匹配长度或上下文长度（越长越大，但应饱和）。
- `f_budget` 来自本 tick CS 预算余量（预算紧张时自然变小）。
- `f_match` 来自弱命中强度 σ（弱命中更小）。
- `f_cp` 来自组分 CP 的折扣（张力越大越不宜被整体吸收遮蔽）。
- `f_part` 来自本次桥接中实际参与对齐的对象数，建议使用饱和增长形式（如 `log(1+n_match)` 归一化），而不是线性奖励长事件。
- `f_drain` 来自同一路径短时抽能疲劳（例如 `1 / (1 + λ_drain * n_recent(sig))`），保证短时间重复抽能只被削弱而不被硬阻断。

### 11.3 ES 能量更新：分别对 ER 与 EV 记账
吸收应分别对 ER 与 EV 执行，避免把“实证据”与“预测”混为一谈。  
对一个拼接动作 `Create/Extend`，设被吸收的组分集合为 `C`，则：
- `ΔER_in = Σ_{i∈C} α_i * ER_i`
- `ΔEV_in = Σ_{i∈C} α_i * EV_i`

同时从组分扣除：
- `ER_i <- (1-α_i) * ER_i`
- `EV_i <- (1-α_i) * EV_i`

引入一个守恒效率参数 `η_cons ∈ (0,1]`（默认建议 0.95~1.0），表示“整体化压缩的散逸损耗”。  
则 ES 获得的能量增量为：
- `ΔER_es = η_cons * ΔER_in`
- `ΔEV_es = η_cons * ΔEV_in`

当 `η_cons < 1` 时，差额能量视为加工成本或散逸，不回流到其他对象。该散逸可作为稳定器，防止事件结构无限硬化。

ES 的初始能量可以取上述增量；也可以叠加一个极小的结构性基底（不建议大，避免凭空造势）。

### 11.4 composition_profile：组分能量比例的定义与更新
ES 必须携带组分能量比例，至少为：
- `p_i^r`：组分 i 的 ER 占比
- `p_i^v`：组分 i 的 EV 占比
并满足分别归一化。

在一次拼接之后，比例更新可用加权平均：
```
p_new = normalize( (1-λ_profile)*p_old + λ_profile*p_obs )
```
其中 `p_obs` 是本次参与拼接的输入能量图景形成的比例，`λ_profile ∈ [0,1]` 由匹配强度与证据可信度决定。

### 11.5 长度治理：用能量阈值推导自然上限
设 ES 的总能量为 `E_total = ER_es + EV_es`，给组分分配实时能量：
`E_i_alloc = E_total * p_i`（可分别对 ER/EV 分配，再合成总量）

若状态池淘汰阈值为 `E_drop`（总能量意义下），则满足：
- 任何组分若 `E_i_alloc < E_drop`，就倾向于在事件收缩时被剔除。

因此在最粗的上界意义下：
`N_max ≤ floor(E_total / E_drop)`  
即：当总能量有限且淘汰阈值固定时，能稳定保留的组分数存在上界。

现实中还有半衰期与容量软上限，使得 `E_total` 随时间衰减，`N_max` 会随时间下降。因此“无限生长”需要持续外部能量注入与持续高 α 吸收，而这两者在资源纪律下都不应长期成立。

### 11.6 状态池软上限按“总长度”计费（可配置）
为防止一个超长 ES 绕开容量限制，建议将状态池的软上限计费对象做成可配置：
- 模式 A：按对象数计费（旧模式）
- 模式 B：按组分数计费（ES 长度也算容量）
- 模式 C：按刺激元展开估计计费（更精细，但成本更高）

落地时建议同时记录两个指标：
- `state_pool_total_component_units`
- `state_pool_total_stimulus_units_est`
默认用“组分数”作为计费治理，必要时可切换为“刺激元数”。

## 12. HDB 联动：ES 数据库的索引、懒融合、闲时巩固与内存释放

### 12.1 ES 数据库索引：链式打开与存在性判定
一种可行的存在性判定方式是“从左到右的链式打开”：
- 打开 `C` 的数据库，查找残差项 `A`，得到 `C+A` 的数据库指针；
- 打开 `C+A` 的数据库，查找残差项 `B`，得到 `C+A+B` 的数据库指针；
- 以此类推，直到判定 `C+A+B+D` 是否存在。

该方法的优点：
- O(length) 的局部打开，不依赖全库扫描；
- 每步都是 O(1) 指针寻址 + top-k 残差查找；
- 与“局部库优先”的工程原则一致。

该方法的局限：
- 若某一步不存在，则只能匹配到最大前缀；
- 对由“带孔子序列”推演出的 ES，链式存在性未必成立，需要懒融合与巩固机制支撑完备性。

### 12.2 懒融合（Lazy Merge）：parents + delta 的 ESDB 表达
为避免每次拼接都立即物化一个独立数据库（导致写放大与膨胀），建议 ESDB 采用懒融合：
- `parents`: 指向一个或多个父数据库（例如 `C+A` 与 `A+B`）
- `delta`: 本 ES 自己新增或更新的残差项（小表）

ESDB 的最小字段集（概念级）建议包含：
- `signature`：`sig_seq(ES)` 与可选 `sig_set(ES)`（用于唯一性与候选定位）。
- `components`：组分结构 id 序列（及其时序组边界）。
- `parents`：父引用列表（每项至少包含 parent_pointer、merge_weight 可选、created_tick、reason）。
- `delta_residual_table`：增量残差表（本 ES 新学到/新写入的边）。
- `stored_profile_mu`：该 ES 的平均能量比例画像（用于评分与弱命中校准）。
- `component_ledger`：组分能量账本（可选但强烈建议；用于中和与裁剪一致性）。
- `materialized`：是否已物理巩固（闲时巩固后置 true）。
- `stats`：base_weight、recent_gain、fatigue、evidence_er_sum、evidence_ev_sum 等统计字段。
- `runtime_cache`：运行时 top-k overlay 合成缓存（必须可清理、可失效）。

运行期打开 ESDB 时：
- 先查 delta；
- 若 delta 不命中，再按父优先级查 parents；
- 对命中项的权重执行融合（见 12.4）。

运行时查询语义建议显式化为 “overlay 合成 + top-k 缓存”：
- 合成时只读取 delta 与每个 parent 的 top-k，不做全量展开。
- 对同一 target 的权重执行融合算子并排序截断。
- 将合成后的 top-k 结果缓存，并以 delta 版本号或 TTL 作为失效条件，避免每 tick 重复合成导致过热。

### 12.3 闲时巩固（Idle Consolidation）：把父链压平并释放内存
懒融合会产生父链，父链若无限增长将导致：
- 打开成本上升；
- 内存与缓存占用增加；
- 最终变成另一个“结构级过热”。

因此必须实现闲时巩固：
- 在空 tick 或低负载 tick 执行；
- 前端提供按钮可手动触发；
- 在跑完数据集时自动触发一次“巩固 + 释放内存”，以便评估长期运行。

触发方式建议至少支持三种（可配置）：
- 手动触发：前端按钮“执行闲时巩固/整理/释放内存”。
- 运行结束触发：一次数据集跑完后自动触发一次（默认开启，便于验收）。
- 长跑低频触发：每 N tick 或当 parent 链深度超过阈值时触发（默认保守关闭或低频）。

巩固的目标：
- 将 overlay 物化为一个独立的 materialized DB；
- 或至少将多层父链压平为 1-2 层；
- 清理中间视图缓存与临时索引；
- 输出可审计统计：巩固了多少 ES、平均父链长度下降多少、释放了多少内存估计等。

巩固的最小工作集建议包含：
- 展开 parents 引用并合成一份物理残差表（对同 target 去重融合）。
- 对物理表做截断：保留高价值 top-N，并保留少量探索尾部（避免过拟合硬化）。
- 必要时重建局部索引（仅限 ESDB 自身，不触发全库重建）。
- 清空 overlay 合成缓存与父表缓存，释放内存。
- 输出巩固报告与 metrics：耗时、处理对象数、去重比、残差条目 before/after、释放缓存估计等。

### 12.4 权重融合算子：避免简单相加导致虚假高权重
若简单相加父库权重，会产生“重复证据”被过度放大问题。建议采用递减收益的融合算子，例如：
`w_merge = max(w1, w2) + β * min(w1, w2)`，其中 `β ∈ [0,1]`。  
当 `β < 1` 时，第二份证据带来较小增益，符合“多源一致性提升但边际递减”的直觉。

此外，需要引入 ER/EV 对权重的调制（本文档额外约束）：
- 实能量对权重增强：`w_r = w * (1 + k_r * ER_norm)`
- 虚能量对权重削弱：`w_v = w * (1 - k_v * EV_norm)`（下限截断到非负）
- 最终有效权重：`w_eff = clamp(w_r - w_v, w_min, w_max)` 或其它可审计形式

该调制的直觉是：
- 被验证的结构连接更可信，应更容易被再次拼接；
- 纯推演的连接应更谨慎，避免推演泡沫写回导致膨胀。

### 12.5 完备性与原子性：两阶段创建 + 修复队列
ES 的创建可能涉及：
- 新对象入状态池；
- ESDB overlay 的创建；
- 索引写入与 delta 写入；
- 未来巩固。

为避免中断导致“半完备 ES”，建议采用两阶段策略：
1. **创建阶段**：先创建 ES 对象与最小 overlay 描述（parents+空delta），标记 `es_state="pending"`  
2. **提交阶段**：写入 delta 与索引，标记 `es_state="ready"`  
若失败则进入修复队列：  
- 修复队列在闲时处理，补齐缺失索引或回滚到父结构。

该策略不依赖硬规则，而是工程一致性保证。

## 13. 缓存中和联动：事件整体与组分的“互补中和”

### 13.1 需求：证据进入时，应能校准事件内部组分的不平衡
当外源刺激提供了对 ES 内部某组分的真实证据时（例如“苹果”真实出现），需要允许：
- 外源刺激中的“苹果”的 ER 对事件内部“苹果”组分的 EV 失衡进行中和；
- 中和结果应降低组分 CP，从而降低事件整体的张力，并改变事件把握感；
- 该中和只能降低认知压，不能提升认知压。

### 13.2 互补中和（Complementary-only Transfer）
设 ES 内某组分 g 具有 `(ER_g, EV_g)`，外源刺激为该组分提供 `(ER_in, EV_in)`。  
互补中和要求只允许“互补方向”的转移：
- 当 `EV_g > ER_g`（虚偏多）时，允许 `ER_in` 转移一部分给组分以缩小差值；
- 当 `ER_g > EV_g`（实偏多）时，允许 `EV_in` 转移一部分给组分以缩小差值。

转移量示例：
`Δ = min( |ER_g - EV_g|, κ * E_in )`
其中 `E_in` 取互补方向能量，κ 为中和系数。

并更新：
- 若虚偏多：`ER_g += Δ`，`ER_in -= Δ`
- 若实偏多：`EV_g += Δ`，`EV_in -= Δ`

该更新会单调降低 `CP_g = |ER_g - EV_g|`，符合“只能降低认知压”的要求。

### 13.3 同方向能量：促进碎片化解释而不是压平事件
若输入能量方向与组分偏差方向相同（例如组分本就虚偏多，又输入更多 EV），则不用于“压平 ES”，而是：
- 留在组分自身对象上（或刺激流对象上），导致其 CP 增加；
- CP 增加会使该组分更容易进入注意力，触发进一步解释或重组；
- 这形成“证据不足时碎片化、证据充分时整体化”的自然动态。

### 13.4 为什么需要 ES 组分账本（否则无法精确中和）
由于 ES 通过能量吸收把组分能量汇聚到整体事件上，组分对象在状态池中的 ER/EV 可能已经被扣减变弱。现实证据到来时，需要中和的往往是“事件内部该组分的失衡”，而不是一个弱小的独立组分对象。

因此 ES 应维护组分账本：
- 对每个组分 i 维护 `(ER_i, EV_i)`。
- 并满足一致性约束：`Σ ER_i = ER_es`、`Σ EV_i = EV_es`。

该账本既服务于中和，也服务于裁剪与把握感计算（内部一致性分量）。

### 13.5 互补中和流程（概念步骤）
当缓存中和阶段处理刺激流并命中某结构 S 时，若 S 同时是若干 ES 的组分，可对这些 ES 执行组分级互补中和（按预算与 σ 缩放）：
1. 定位组分  
确定命中的结构 S 在 ES 的哪个组分槽位（可能多个 ES、多处出现）。
2. 计算互补补足量  
若 `EV_i > ER_i`，只允许使用输入中的 `ER_in` 补足差额；若 `ER_i > EV_i`，只允许使用输入中的 `EV_in` 补足差额。
3. 记账式能量转移  
从输入能量池扣减补足量，并写入 ES 组分账本；同时更新 ES 总能量（或在本 tick 末统一汇总）。
4. 约束：只降不升  
补足量不得超过差额，否则会反向制造差异；中和只能降低 `|ER_i - EV_i|`。
5. 同向能量不用于压平  
同向 EV/ER 不参与 ES 组分中和，保留在碎片对象上以促使解释与重组。

### 13.6 中和联动观测字段（建议）
为避免“中和写了但不知是否生效”，建议新增至少以下观测字段：
- `event_component_neutralization_count`
- `event_component_neutralization_er_added_sum`
- `event_component_neutralization_ev_added_sum`
- `event_component_cp_drop_sum`
- `event_component_neutralization_budget_hit_flag`

## 14. 新认知感受：事件把握感（Event Grasp）

### 14.1 定义
事件把握感是一种属性刺激元（绑定到 ES 对象），以数值刺激元形式表达“对该事件整体的把握程度”。它与违和感、置信度等认知感受同类处理，进入桶数据库并参与后续联动。

### 14.2 触发条件（尽量少硬规则）
事件把握感不应对所有 ES 常驻发射。建议触发条件为：
- ES 的总能量高于阈值（表示事件在当前确实“存在感强”）
- ES 被注意力选入 CAM（表示本 tick 真正注意到了它）

这两个条件是最小且语义清晰的约束，避免把握感在低能量事件上泛滥。

### 14.3 数值计算因素（可调参）
把握感数值可以由以下因素构成：
- `E_total`：事件总能量（对数压缩）
- `balance`：事件内部平衡度（组分 CP 的聚合，例如 `1 - mean(CP_g_norm)`）
- `margin`：事件相对竞争优势（与 runner-up 的分差）
- `order_evidence`：顺序证据（γ 与 σ 的函数）

示例：
`grasp = sigmoid( a*log(1+E_total) + b*balance + c*margin + d*order_evidence )`

更可解释的一种分解形式是把把握感写成若干分量的加权和，再压到 `[0,1]`：
- 证据比例分量：`r = ER_es / (ER_es + EV_es + ε)`
- 领先优势分量：`m = clamp((s1 - s2) / (|s1| + ε), 0, 1)`（s1/s2 为事件竞争分数前二名）
- 内部一致性分量：`c = 1 - normalize(mean_g CP_g)`（由组分账本计算）
- 顺序证据分量：`t = γ` 或 `t = match_sigma` 的统计量

然后：
`grasp = sigmoid( a*(w_r*r + w_m*m + w_c*c + w_t*t) + b )`

其中 `a,b,w_*` 都应进入配置与 APT 可调参空间，使“把握感强度”可随系统风格与数据集分布长期标定，而不依赖固定硬规则。

### 14.4 示例：紫苹果
初期由“紫色 + 苹果”拼接形成 ES，若主要来自推演 EV，则：
- `E_total` 可能不低，但 `balance` 差（苹果可能虚偏多），把握感中等或偏低；
后续外源刺激真实看到了苹果：
- 互补中和降低苹果组分 CP，使 `balance` 上升；
- ES 把握感随之上升；
若后续又出现更多不确定性推演：
- 同方向 EV 不用于压平 ES，反而使“苹果”碎片对象更易被注意，从而触发重解释。

### 14.5 表达为数值刺激元并桶化（与既有认知感受一致）
事件把握感应以“属性刺激元 + 数值刺激元”的方式进入既有感受系统：
- 绑定对象：ES（事件结构）作为锚点对象。
- 刺激类型：数值刺激元（value ∈ [0,1]）。
- 桶化：映射到 numeric bucket（线性分段或非线性分段均可，优先保证可解释与稳定）。
- 写回：作为本 tick 的属性刺激元进入状态池（并可按既有规则写入桶数据库以参与长期统计）。

触发仍建议保持克制（第 14.2 节），避免把握感在低能事件上泛滥，从而污染长期观测与调参。

## 15. 可观测性与前端交互（控制面板、长期图表、审计字段）

### 15.0 单次 Tick 观测：必须能回答的 8 个问题
认知拼接引入“事件结构进入状态池”，这会显著增强拟人性，但也会显著提高系统解释难度。因此单次 Tick 观测台必须能清晰回答以下问题，否则调参与验收会退化为猜测：
- 本 tick 的主叙事 ES 是什么？新建了哪些 ES？扩展/前置了哪些 ES？裁剪/收缩了哪些 ES？
- 每个 ES 的组分序列是什么？时序视图与共现视图分别是什么？
- 每次拼接使用了多长上下文（k 阶后缀/前缀）？是否使用带孔子序列？`gap_count` 是多少？
- 拼接吸收了多少能量（ER/EV 分别多少）？被吸收组分对象能量变化是什么？
- ES 的 `composition_profile` 与组分账本如何变化？哪些组分占比上升/下降？
- 弱命中发生了多少次？`σ_seq`、`σ_set`、`γ` 的贡献分别如何？
- 缓存中和阶段对 ES 组分发生了多少次互补中和？降低了多少组分 CP？
- 是否触发闲时巩固？巩固了哪些 ES？父链长度下降多少？释放缓存估计多少？

为了满足上述问答，CS 输出的 `cs_report` 建议分为三层：
- `brief`：本 tick 创建/扩展/裁剪数量、耗时、吸收能量等摘要。
- `detail`：逐条候选与胜出动作的评分分解、上下文、弱命中分解、停止原因。
- `trace`：与 AF、中和、刺激级查存一体、赋能、CFS/情绪/行动串联的 `trace_id`。

### 15.1 CS 模块必须输出的关键指标（每 tick）
建议至少新增如下 metrics（字段名可按通用字段标准调整）：
- `cs_enabled`（0/1）
- `cs_state_pool_es_count`：状态池内 ES 数量
- `cs_state_pool_es_total_component_units`：ES 组分总数（所有 ES 的组分数求和）
- `cs_state_pool_es_total_stimulus_units_est`：ES 展开刺激元估计总量
- `cs_stitch_attempt_count`：本 tick 尝试拼接候选数
- `cs_stitch_success_count`：本 tick 成功拼接次数
- `cs_stitch_best_score`：最佳拼接得分
- `cs_context_len_used_hist`：使用的上下文长度分布（可简化为均值/最大/分位）
- `cs_weak_hit_sigma_mean`：弱命中 σ 均值
- `cs_gamma_mean`：顺序证据 γ 均值
- `cs_gap_penalty_mean`：gap 惩罚均值（若启用）
- `cs_energy_absorb_er_sum / cs_energy_absorb_ev_sum`：吸收能量总量
- `cs_component_pruned_count`：本 tick 被剔除组分数
- `cs_es_pruned_count`：本 tick 被收缩/重建的 ES 数
- `cs_timing_ms`：CS 总耗时
- `cs_narrative_top_energy_share`：主叙事 ES 在全部 ES 能量中的占比
- `cs_narrative_top_grasp`：主叙事 ES 的事件把握感
- `cs_recent_stitch_log_count`：最近拼接记录条数
- `cs_narrative_switch_count`：滚动窗口内主叙事切换次数

- `cs_idle_consolidation_triggered`（0/1）

### 15.3 前端控制面板（Observatory）
前端至少提供：
- `enable_cognitive_stitching` 开关
- `enable_structure_level_retrieval_storage` 开关
- CS 参数面板（top-k、吸收比例范围、γ 计算参数、gap 惩罚、长度治理模式等）
- “闲时巩固”按钮（触发一次 consolidation）
- 状态池叙事化视图：按“总能量 top”“把握感 top”“当前竞争第二名”展示主事件对象
- 最近拼接记录时间线：Create / Extend / Prepend / Merge / Prune 的最近动作流
- ES 列表视图（可展开查看 components、composition_profile、energy、最近拼接审计）
- 事件内部热点视图：展示 ES 内高 CP 组分、当前薄弱组分与中和命中位置
- 事件把握感图表（本 tick、滚动窗口、分布）

叙事化视图不应只显示“能量最高的 ES”。更稳妥的做法是至少并列展示三类榜单：
- 总能量 top：谁当前最占资源。
- 把握感 top：谁当前最像主叙事。
- 高张力热点：哪个 ES 虽未必能量最高，但内部冲突最值得解释。

### 15.4 长期运行页面（Long-run Dashboard）新增图表
建议至少新增：
- ES 数量与总长度（component_units 与 stimulus_units_est）
- CS 耗时与拼接成功率
- γ 与 σ 的均值/分位变化（观察弱命中是否长期退化）
- 事件把握感峰值与均值（与违和/置信度对照）
- 闲时巩固触发次数与效果（父链长度下降、内存估计释放）
- 主叙事切换次数与主叙事能量占比（观察是否长期被单一叙事垄断）
- 最近拼接动作分布（Create/Extend/Merge/Prune 的比例变化）

### 15.5 UI 交互规范（避免页面过长、支持放大、支持巩固操作）
为保证观测体验可用且可验收，前端建议遵循以下最小规范：
- 长列表必须使用固定高度滚动容器（候选动作列表、ES 列表、规则列表、残差表展开等）。
- 模块分区应支持折叠/展开，标题行显示关键摘要（例如本 tick ES 数、CS 耗时、巩固触发标记）。
- 图表应支持点击放大（弹窗/抽屉），放大视图展示更多统计与关键事件标注。
- 闲时巩固按钮应提供进度与日志窗口，并输出巩固审计报告（处理对象、去重比、释放缓存估计、耗时）。

## 16. 自适应调参系统（APT）联动

### 16.1 可调参数建议（示例）
CS 相关参数建议分为几类：
- 候选宽度：
  - `cs.residual_top_k_per_node`
  - `cs.max_stitch_attempts_per_tick`
- 弱命中：
  - `cs.gamma_input_base_external`
  - `cs.gamma_input_base_internal`
  - `cs.gamma_target_order_sensitivity_scale`
- 上下文回退：
  - `cs.max_suffix_context_len`
  - `cs.max_prefix_context_len`
  - `cs.allow_anchored_subsequence`
  - `cs.gap_penalty_lambda`
- 能量账本：
  - `cs.absorb_alpha_min/max`
  - `cs.absorb_k_w/k_m/k_c/k_p`
  - `cs.component_drop_energy_threshold`
  - `cs.capacity_accounting_mode`（object/component/stimulus）
- HDB 懒融合与巩固：
  - `cs.overlay_parent_limit`
  - `cs.idle_consolidation_enable`
  - `cs.idle_consolidation_interval_ticks`
  - `cs.idle_consolidation_budget_ms`
- 中和联动：
  - `cs.neutralization_kappa`

### 16.2 常态值哲学：多数感受与奖惩应处于半量程以下
对于奖惩信号、认知感受以及情绪递质等“强度型变量”，APT 的理想值不宜默认贴近 0.5 以上。更稳妥的总原则是：
- 大多数普通时刻，这些变量应处于半量程以下；
- 常态理想值通常可落在 `0.3` 左右；
- 常见稳态带可粗略落在 `0.1~0.5`；
- 明确的外界强信号、教师奖励/惩罚或显著违和事件，才更适合把数值推到 0.5 以上。

这一原则的价值在于保留“正常”与“强烈”的可分性。若常态长期贴近高位，系统会失去显著峰值空间，APT 也会更难区分普通波动与真正异常。

但该原则应被理解为“分布哲学”，而不是“所有指标统一等于 0.3”。具体落地时应区分变量类型：
- 奖惩、违和、压力、期待、把握感这类强度型变量，适合低于半量程的常态分布。
- 动作驱动力、对象数、耗时等并不都适用同一个理想点。
- 不同递质通道的半衰期与基线也不完全相同，应按通道语义分别标定。

因此，APT 更应关注以下统计量，而不只是一个单点 ideal：
- 均值/中位数：常态是否偏低于 0.5；
- 高位占用率：`value > 0.5` 的时间比例是否只在强事件时显著上升；
- 分位数：如 p90/p95 是否保留峰值空间；
- 半衰期：强脉冲是否能在合理 tick 数内回落，而不是长期高位驻留。

### 16.3 APT 的规则建议（面向稳态异常而非结果指标）
建议 APT 的规则关注“上游可控点”，避免把结果指标当控制点：
- 若 `cs_timing_ms` 超标：优先下调 residual top-k、max attempts、max context len
- 若 `cs_state_pool_es_total_component_units` 过高：提高 drop threshold 或降低吸收比例上限（让事件更难长时间维持）
- 若 `cs_stitch_success_count` 长期为 0：适当放松 top-k 或降低 gap 惩罚（鼓励形成事件）
- 若 `event_grasp` 长期过低但 ES 很多：降低吸收比例或提高中和系数（让事件更易平衡）
- 若 ES 过于垄断注意力：提高 conflict penalty 或降低 margin 增益项

### 16.4 参数-指标映射表（最小可审计版）
为了让 APT 的动作可解释且可回滚，建议维护一份“参数-指标映射表”，至少覆盖 CS 的上游可控点。示例（非穷尽）：

| 参数 | 主要影响的指标 | 典型风险 |
| --- | --- | --- |
| `cs.residual_top_k_per_node` ↑ | `cs_stitch_attempt_count` ↑，`cs_timing_ms` ↑ | 候选爆炸、过热 |
| `cs.max_stitch_attempts_per_tick` ↑ | `cs_stitch_success_count` ↑，`cs_timing_ms` ↑ | 事件过度生成 |
| `cs.max_suffix_context_len` ↑ | `cs_context_len_used_*` ↑，命中质量可能 ↑ | 回退搜索成本上升 |
| `cs.gap_penalty_lambda` ↑（惩罚更弱） | `cs_gap_count_mean` ↑，事件更长 | 错误叙事风险上升 |
| `cs.absorb_alpha_max` ↑ | ES 更稳、碎片更弱 | 叙事硬化、垄断风险 |
| `cs.component_drop_energy_threshold` ↑ | `cs_component_pruned_count` ↑，事件更短 | 碎片化过强 |
| `cs.overlay_parent_limit` ↓ | overlay 更浅、查询更快 | 学习速度下降、过度截断 |
| `cs.neutralization_kappa` ↑ | `event_component_cp_drop_sum` ↑ | 中和过强可能遮蔽矛盾 |

### 16.5 规则簇与回滚判据（建议先落地的 6 类）
为避免规则爆炸，建议先按“稳态异常”建立少量规则簇，并为每类定义回滚判据：

- 事件过长或总容量过高  
触发：`event_len_components_p95` 或 `state_pool_total_component_units` 超范围且趋势上升。  
动作：上调 `cs.component_drop_energy_threshold`，下调 `cs.absorb_alpha_max`，或下调 `cs.max_suffix_context_len`。  
回滚：ES 长度被压到几乎常数且 `cs_stitch_success_count` 贴近 0（事件层失活）。

- 事件过短、碎片过多  
触发：`cs_state_pool_es_count` 很低但状态池对象总量高、候选过宽指标上升。  
动作：上调 `cs.absorb_alpha_max` 或上调 `cs.residual_top_k_per_node`（小步），并降低裁剪阈值。  
回滚：`cs_timing_ms` 明显恶化或 ES 垄断注意力。

- 弱命中过强导致“共现当顺序”  
触发：`cs_weak_hit_sigma_mean` 偏高且 `cs_gap_count_mean` 上升，同时违和/压力尖峰增多。  
动作：提高 `os(X)` 对顺序失败的敏感度（或下调 `σ_set` 上限折扣），并提高 gap 惩罚。  
回滚：弱命中全部消失导致内源刺激几乎无法命中事件结构。

- 懒融合父链过深  
触发：`cs_idle_consolidation_parent_chain_before_mean` 持续上升且查询耗时上升。  
动作：提高巩固频率或 batch size，或降低 `cs.overlay_parent_limit`。  
回滚：巩固耗时挤占主流程预算。

- 中和联动无效  
触发：`event_component_neutralization_count`、`event_component_cp_drop_sum` 长期近零且 CP/违和长期高位。  
动作：上调 `cs.neutralization_kappa` 或提高“命中组分->中和”的优先级（仍按 σ 缩放）。  
回滚：中和过强导致矛盾被压平，事件把握感长期贴顶。

- CS 过热  
触发：`cs_timing_ms` 超标并与候选数共振。  
动作：下调 `cs.residual_top_k_per_node`、`cs.max_stitch_attempts_per_tick`、`cs.max_suffix_context_len`，并启用/加强“预算熔断 stop_reason”。  
回滚：事件层失活（成功拼接长期为 0）。

## 17. 数据集：拟人陪伴 Bot 对话输入模拟（小/中/大规模 + 空 tick 时间间隔）

### 17.1 数据集设计目标
为了更真实评估“持续认知系统”的稳态行为，需要一个更贴近部署环境的数据集。该数据集应模拟：
- 用户输入（多主题、偶发高情绪、日常碎片）
- 大模型回复（可带工具调用、解释、安抚、计划）
- 系统信息（会话开始/结束、权限、网络波动、提醒）
- API 返回（天气、日程、定位、搜索、设备状态等）
- 操作信息（点击、打开页面、失败重试、确认/撤销）
- **空 tick**：代表现实中“没有新输入但系统仍在运行”的时间，默认 1 个空 tick 对应 3 秒现实时间

空 tick 的价值在于：
- 允许内源性活动在“无外源主导”时发生；
- 允许虚能量传播与回忆成为主导，形成更拟人的“自己在想”；
- 为闲时巩固提供自然触发窗口。

### 17.2 基本数据模型（建议）
每个 tick 记录一个或多个输入事件，事件类型可包括：
- `user_message`
- `assistant_message`
- `system_event`
- `api_request`
- `api_response`
- `operation_event`
- `idle_tick`（空 tick）

并携带：
- 时间戳（或相对时间）
- 线程/会话 id
- 文本内容或结构化 payload（API 返回、操作结果）
- 可选：情绪标签或意图标签（用于评估，不用于硬规则驱动）

### 17.3 时间尺度与规模划分
假设：1 tick 的“空 tick 间隔”默认对应 3 秒现实时间。  
则：
- 20 tick 约 1 分钟  
- 1,200 tick 约 1 小时  
- 28,800 tick 约 1 天

据此定义三种规模：
1. 小规模：`~1,200 tick`（约 1 小时）  
   目标：快速迭代、调参与功能验证（ES 是否生成、把握感是否合理、巩固是否可触发）。  
2. 中等规模：`~7,200-14,400 tick`（约 6-12 小时）  
   目标：观察稳态收敛、候选空间是否膨胀、ES 长度分布、空 tick 内源性占比。  
3. 大规模：`~57,600-86,400 tick`（约 2-3 天）  
   目标：验证长时间运行的资源纪律、HDB 增长曲线、闲时巩固有效性与内存释放效果。  

规模可按工程成本调整，但应保持时间语义一致。

### 17.4 场景与内容分布建议
数据集应包含多类“拟人陪伴”典型片段，并穿插空 tick：
- 日常寒暄与陪伴：问候、关心、分享日常
- 任务型：查天气、记日程、提醒、查询信息
- 情绪段：焦虑、误会、吃醋、不回信息的解释竞争
- 反事实与否定：如果…会怎样、不是…而是…
- 连续多轮对话：上下文持续
- 突发事件：网络失败、API 超时、权限拒绝
- 睡眠/离线段：长时间 idle_tick，让系统以虚能量为主运行

### 17.5 可复现实验：种子与脚本化生成
为保证验收严格，建议数据生成支持：
- 固定随机种子
- 可配置的“日程模板”与“主题分布”
- 可插拔的 API 模拟器（天气、日历、搜索等）
- 输出格式统一（例如 jsonl）

### 17.6 输入事件字段建议（面向“部署在聊天软件上的陪伴 Bot”）
为了让数据集更像真实部署环境，建议每个 tick 的事件条目至少包含：
- `event_type`：user_message / assistant_message / system_event / api_request / api_response / operation_event / idle_tick
- `timestamp` 或 `t_rel_sec`：相对时间秒数（空 tick 默认为 +3s）
- `session_id`、`thread_id`：多会话与跨线程模拟
- `role`：user / assistant / system（对 message 类事件）
- `text`：文本内容（对 message 类事件）
- `payload`：结构化内容（对 API/操作/系统事件）

对 API 事件建议补充：
- `api_name`、`request_id`、`status`（ok/timeout/denied/error）
- `latency_ms`（用于模拟现实中等待与失败重试）
- `result_summary_text`（可选：将结构化结果转为可读摘要，便于系统进入刺激流）

对操作事件建议补充：
- `operation`（click/open/confirm/cancel/retry/scroll 等）
- `target`（目标对象或页面）
- `result`（success/fail + 原因）

### 17.7 训练样本中的“期望契约”与延迟结果监督
为了让训练样本更接近真实交互，不应只支持“当前 tick 立即给奖励/惩罚”的静态标签，还应支持一种带时窗的期望契约（expectation contract）：
- 某个输入 tick 创建一个待满足的期望；
- 系统在后续 `N` 个 tick 内观察动作、工具调用、状态变化或文本反馈；
- 若满足成功条件，则注入奖励并生成“执行成功”类反馈；
- 若超时或触发失败条件，则注入惩罚并生成“没有执行”类反馈。

该机制的关键不是改写原始输入，而是在未来 tick 插入新的系统反馈事件。也就是说，原始输入保持不可变、可审计；成功或失败应作为新增的 `system_event` 或 `teacher_feedback_event` 写入时间线上，而不是回头篡改历史文本。

建议的数据字段至少包括：
- `expectation_id`
- `window_ticks`
- `anchor_policy`（奖励/惩罚最终回绑给谁）
- `success_conditions`
- `failure_conditions`
- `on_success`
- `on_failure`
- `emit_feedback_event`（是否生成可读反馈事件）

其中：
- `success_conditions` 与 `failure_conditions` 应优先支持结构化条件，如 action/tool/state/metric/attribute，而不仅是文本包含。
- `anchor_policy` 必须尽量绑定到“创建期望时的责任对象或责任事件”，避免若干 tick 之后把奖励/惩罚错误发给当前 top1。
- `on_success` / `on_failure` 可同时包含数值奖惩与文本反馈模板，但文本模板应视为环境反馈，而非唯一监督信号。

这一设计与现有 teacher feedback 机制兼容：立即奖惩仍可保留为最小闭环，而“期望契约”则作为更真实的大规模训练协议，用于评估执行、延迟结果与自我纠偏能力。

### 17.8 空 tick 与“睡眠段/离线段”的建模
空 tick 的设计目的不是“填充数据”，而是引入真实部署中的时间结构，使系统获得：
- 没有外源输入时的内源性持续认知窗口（EV 传播、回忆、事件巩固）。
- 闲时巩固的自然触发窗口（避免把巩固挤进高负载 tick）。

建议支持两类 idle：
- `idle_tick`：单个空 tick，默认 +3 秒。
- `idle_span`（可选）：压缩表示一段连续 idle（例如 10 分钟），生成时可展开为多个 idle_tick，或在运行时按 span 处理以节省 IO。

### 17.9 输出格式建议（jsonl，按 tick 组织）
建议每行一条 tick 记录，包含：
- `tick_id`
- `t_rel_sec`
- `events`: 本 tick 的事件列表（可为 0 条，用于 idle）
- `meta`: 可选元信息（场景段标签、随机种子片段、注入的失败模式等）

该格式的优势是：
- 可自然表达“一个 tick 内多个事件”（例如 api_request 与 api_response 分属不同 tick）。
- 易于回放、重放与做滚动窗口统计。

### 17.10 数据集验收点（与 CS/结构级改造直接相关）
为了验证“空 tick + 多通道输入”确实改善系统稳态，建议数据集验收至少包含：
- 空 tick 占比是否达到预期（例如 40%~80%，按规模与场景而定）。
- 内源性刺激占比是否提升（长期运行中多数 tick 内源刺激应显著多于外源）。
- 虚能量链路是否得到窗口（空 tick 段 EV 增量应可观测，而非永远被 ER 主导）。
- 闲时巩固是否可在 idle 段触发且不挤占主流程预算。
- 延迟结果监督是否能在窗口内完成判定，且不改写原始输入事件。
- 奖励/惩罚是否回绑到期望创建时的责任对象，而不是事后漂移到任意高能对象。

## 18. A/B 实验与验收标准

### 18.1 必做对照
为保证可回滚与严格验收，建议至少覆盖如下开关矩阵（按项目配置字段命名）：
- Baseline-A：`enable_structure_level_retrieval_storage=0`，`enable_cognitive_stitching=0`（无事件层基线）
- Baseline-B：`enable_structure_level_retrieval_storage=1`，`enable_cognitive_stitching=0`（旧结构级事件层）
- CS-only：`enable_structure_level_retrieval_storage=0`，`enable_cognitive_stitching=1`（新事件层默认路线）
- Hybrid：`enable_structure_level_retrieval_storage=1`，`enable_cognitive_stitching=1`（叠加态，仅实验）

### 18.2 资源纪律验收（硬指标）
至少覆盖：
- `timing_structure_level_ms`：新机制下应显著下降或不再是主耗时
- `timing_total_logic_ms`：整体应下降或更稳定
- `attention_state_pool_candidate_count`：应下降或更可控
- `hdb_structure_count` 增长速率：应下降或趋稳（不要求立刻下降）
- `cs_timing_ms`：CS 自身耗时必须在预算内

### 18.3 逻辑效果验收（软指标 + 观测）
关注：
- ES 是否形成（不是全 0）
- ES 是否能被注意力选中（不是永远弱势）
- 事件把握感是否与证据变化一致（苹果被证实后把握感上升）
- 证据不足时是否倾向碎片化解释（同方向能量导致组分 CP 上升）
- 闲时巩固是否有效降低父链长度并释放缓存
- 延迟结果监督是否能正确区分“执行成功”与“未执行”，并把奖惩注入到与原期望一致的责任对象/事件上

### 18.4 失败判据（出现即需要回滚或优先调参）
下列现象若稳定出现，应视为“当前配置或实现不满足资源纪律/可解释性”：
- `event_len_components_p95` 持续右移且裁剪率不响应（事件长度尾部失控）。
- 懒融合 parent 链深度持续上升且巩固无效（存储侧过热）。
- `event_grasp` 长期贴 1.0 或贴 0.0（直线化或失灵，失去拟人价值）。
- `cs_timing_ms` 成为新的主耗时黑洞且无法通过候选收紧与巩固回收。
- `event_component_cp_drop_sum` 长期为 0 且 CP/违和长期高位（中和联动未生效，整体化无法随证据稳定）。

### 18.5 通俗示例（用于直觉理解，不代表唯一实现）
示例 1：紫苹果（想象->确信->验证->稳定）
- 初期由共现弱命中拼接得到 ES，EV 占主导，把握感中低。
- 外源“苹果”出现时对 ES 内苹果组分执行互补中和，降低组分 CP，把握感上升。
- 若后续只叠加同向 EV，不用于压平 ES，反而促使“苹果碎片”更易被注意，触发重解释。

示例 2：查询天气->城市->太原（内心独白骨架增长）
- `[查询,天气]` 形成后，感应赋能使“城市”获得 EV，事件扩展为 `[查询,天气,城市]`。
- 多个城市候选竞争，历史共现与用户相关性使“太原”胜出，事件扩展为更可读的骨架。

示例 3：不回信息（出轨 vs 很忙）的事件切换
- 初期某一路径 ES1 可能因经验导致高 EV 并成为主叙事。
- 新证据进入后，ES2 被实证据增强而 ES1 被虚证据磨损回收，事件竞争切换主解释。

## 19. 风险清单与缓解策略

### 19.1 事件垄断与叙事单一化
风险：某个高能量 ES 长期垄断注意力，压制其它解释。  
缓解：能量衰减 + 软上限计费 + conflict penalty（连续）+ 边际递减融合算子。

### 19.2 推演泡沫写回导致 HDB 膨胀
风险：弱证据推演的 ES 被大量写回，残差表膨胀。  
缓解：EV 削弱权重 + 低 σ 时 α 低 + 懒融合 + 闲时巩固审计 + overlay 父链上限。

### 19.3 完备性破坏（半结构、不可索引）
风险：中途失败产生不可打开的 ESDB。  
缓解：两阶段创建 + 修复队列 + 只允许 ready 的 ES 参与关键写回。

### 19.4 性能回退（CS 成为新的过热中心）
风险：候选宽度或上下文回退过大导致 CS 过热。  
缓解：严格 top-k + attempts cap + 审计字段 + APT 规则以可控点收敛 + idle consolidation 降父链成本。

### 19.5 延迟监督的奖惩误绑与文本监督投机
风险：若“执行成功/没有执行”类反馈只靠文本回写，或奖惩在超时后临时绑定当前 top1，对象归因会漂移，系统可能学会“说自己做了”，而不是学会真正完成执行。  
缓解：使用结构化 success/failure 条件；期望创建时固定责任锚点；把反馈作为新的系统事件写入，而不是篡改原始输入；将工具/action trace 作为监督的优先证据。

## 20. 落地路线图（只规划，不执行）

落地遵循“先可回滚再增强”的工程节奏：
1. 增加两个开关（结构级开关、CS 开关），并在 tick 主链加入空实现 CS（不改变行为）  
2. 建立 ES 对象与最小 state_pool 进入/淘汰语义（能量账本先以最简单形式实现）  
3. 实现基础拼接：仅用端点残差 `last -> next` 与 `prev -> first`（无回退）  
4. 引入上下文回退与弱命中，并加入审计字段与长期图表  
5. 引入懒融合 ESDB（parents+delta），并实现闲时巩固按钮与自动巩固  
6. 引入缓存中和联动到 ES 组分（互补中和）  
7. 引入事件把握感（IESM 属性刺激元 + 前端展示 + 长期图表）  
8. 在观测台加入“状态池叙事化视图”：展示主叙事 ES、最近拼接记录、事件竞争切换与事件内部高 CP 热点  
9. 接入 APT 新参数与规则，开始小规模数据集调参，并按“常态低于半量程”的哲学重新标定奖惩/感受类理想区间  
10. 在实验 runner 层加入带时窗的期望契约与延迟结果监督，用于工具执行、任务完成与失败反馈的真实训练  
11. 建立小/中/大规模拟人 bot 数据集生成器，做稳态验收与性能剖析  
12. 再考虑结构级查存一体的角色调整：改为闲时深检索或低频巩固，而非每 tick 常驻

```python
def cognitive_stitching_tick(state_pool_snapshot, hdb, cfg, now_tick):
    if not cfg.enable_cognitive_stitching:
        return [], {}, {}

    nodes = collect_active_nodes(state_pool_snapshot)  # ST + ES
    candidates = []

    for node in nodes:
        # Generate candidate edges from local residual top-k
        res = hdb.open_local_db(node.hdb_pointer).get_residual_topk(cfg.residual_top_k_per_node)
        candidates.extend(build_edge_candidates(node, res, cfg))

        # Context backoff: for ES also expand suffix/prefix contexts
        if node.type == "ES":
            candidates.extend(build_context_backoff_candidates(node, hdb, cfg))

    scored = score_candidates(candidates, state_pool_snapshot, cfg)
    winners = select_winners(scored, cfg)  # competition with caps

    es_updates = apply_winners_with_energy_ledger(winners, state_pool_snapshot, hdb, cfg)
    es_updates = prune_and_repair(es_updates, state_pool_snapshot, hdb, cfg)

    report = build_audit_report(winners, es_updates, cfg)
    metrics = build_metrics(report, es_updates, cfg)
    return es_updates, report, metrics
```

## 结束语
认知拼接并不是对既有查存一体理论的否定，而是将“事件组织”从高成本的结构级路径中解耦出来，交由能量、竞争、弱命中与闲时巩固共同塑形。它强调可回滚、可观测与资源纪律，使系统在长时运行中更接近拟人陪伴机器人的稳态输入分布，并为后续实验提供更公平的评估基础。
