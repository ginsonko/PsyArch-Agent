# 行动管理模块（Action/Drive）设计文档（模块 7）

生成时间：2026-04-12  
理论依据：`txt版本的理论核心.txt`（重点参考：3.12 Step 9、4.2.1、4.2.2）

> 行动系统在 AP 中不是外部黑箱，而是“行动节点—驱动力—行动器”的三层结构：行动意图可进入状态池并被记忆、被赋能、被竞争，最终由行动器执行并将反馈回流进入认知闭环。

---

## 1. 模块定位与职责边界

### 1.1 理论定位（4.2.1）
行动系统三层：
1. **行动节点（Action Node）**：一种特殊认知对象，可进入 SP、可被写入结构、可被回忆再激活、可参与行动竞争。
2. **驱动力（Drive）**：行动从“被想到”到“被执行”的中间变量，用于把“认知显著性”与“执行倾向”解耦。
3. **行动器（Actuator）**：外部执行接口（语言输出、工具调用、视觉焦点移动、具身动作等）。

### 1.2 工程职责
行动管理模块应负责：
- 维护行动节点运行态：候选集合、疲劳/近因、绑定情境参数
- 计算并更新 Drive（每 tick）
- 对候选行动做竞争排序与阈值判断
- 执行行动器调用，并处理成功/失败反馈
- 将反馈编码为可回流的刺激元/结构（外源反馈）

### 1.3 不做什么
- 不替代 HDB 的结构形成与记忆组织
- 不替代 EMgr 的情绪调制（行动只消费调制参数）
- 不把行动“直接写死成 if-else 规则”（先天触发应放在 IESM）

## 2. 行动节点（Action Node）的工程表达

### 2.1 行动节点的认知地位（4.2.1.1）
行动节点可以：
- 进入状态池
- 成为结构的一部分被长期记忆记录
- 在匹配、赋能、传播与回忆过程中被重新激活
- 参与认知感受与情绪调制
- 进入行动竞争过程

典型行动节点包括：
- 语言输出类行动
- 工具调用类行动
- 视觉焦点移动行动
- 回忆行动（时间感受器触发）
- 注意力聚焦/发散行动
- 停止/取消类行动

### 2.2 行动节点与“行动反馈结构”的区分（4.2.2.1）
- 行动节点本体：表示“执行某行动”的内部节点
- 行动反馈结构：表示“某行动已经发生”的认知结果

工程上建议两者 object_type/sub_type 不同，避免把“已发生事实”误当成“可执行意图”。

## 3. Drive 的更新与竞争机制

### 3.1 Drive 的输入因子（4.2.1.2）
驱动力更新至少要考虑：
1. 行动节点当前所获能量
2. 结构中的奖励/惩罚底色
3. 当前期待、压力、违和感、正确感等 CFS
4. 情绪递质通道状态（NT）
5. 近因增益与疲劳状态
6. 行动器对应的实时阈值

### 3.2 行动阈值与执行结果（4.2.1.4）
当 Drive 超过实时阈值时尝试触发行动器：
- 成功：按阈值量消耗 Drive，并获得反馈
- 失败：Drive 可保留为意图而不扣除，失败信息作为反馈进入系统，条件变化后可重试

### 3.3 建议的工程公式（可脚本化）
理论未强制给出唯一公式，建议先用可审计分解式：
- `Drive_next = clamp( Drive_prev * decay + Gain(energy, rwd_pun, cfs, nt) - Cost(fatigue, risk), 0, Drive_max )`
- `Threshold_rt = base_threshold * Scale(nt, mode) * Scale(fatigue, recency, action_kind)`

并在 observatory 中输出 Gain/Cost/Threshold 的分解项，方便调试。

## 4. 行动器接口设计

### 4.1 语言行动器（对话型原型）
输出：一段文本（或结构化消息）。  
输入：当前 CAM/关键结构摘要、CFS 摘要、情绪调制参数、选中的行动节点及其情境参数。

> 重要：语言生成本身可以由外部 LLM 实现，但行动管理模块负责“何时输出/输出什么意图”，而不是“怎样写句子”。

### 4.2 工具行动器（Tool-Agent 扩展）
输入可包含：工具名、参数、预算（时间/费用）、安全策略。  
反馈应编码为外源刺激（结构化 SA/CSA），回流到 text_sensor/state_pool。

## 5. 与其他模块的分工协作

### 5.1 与 AF（注意力）协作
- 注意力本身也是行动器（内在行动器）：可由 IESM/Drive 触发“聚焦/发散”
- 行动模块可向 AF 输出 `focus_directives`（带参聚焦），供下一 tick 注意力增益

### 5.2 与 CFS（认知感受）协作
- CFS 为 Drive 更新提供关键调制（违和、惊、期待、压力、正确事件）

### 5.3 与 EMgr（情绪递质）协作
- EMgr 输出 NT 与 Rwd/Pun 底色，影响 Drive 增益、阈值缩放与重试倾向

### 5.4 与 IESM（脚本）协作
- “先天触发规则”放在 IESM：当满足条件时给行动节点赋能/带参触发
- 行动模块执行时应记录脚本版本与触发链，便于审计

---

## 6. 现阶段接入策略（建议）

当前原型尚未实现行动模块。建议 MVP 接入顺序：
1. 先实现 **行动节点对象模型 + Drive 数值维护**（不执行外部行动器）
2. 仅在观测台展示候选行动节点、Drive、阈值与竞争排序
3. 再接入一个最小语言行动器（例如固定模板输出或调用外部 LLM）
4. 最后接入工具行动器与失败重试策略
