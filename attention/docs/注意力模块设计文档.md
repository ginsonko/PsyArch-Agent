## 1. 模块定位与职责边界

### 1.1 理论定位（必须对齐）
理论核心对 AF/CAM 的定义（3.3.1）要点如下：
- **AF（注意力滤波器）**：在预算约束下对 **SP（状态池）对象**做筛选、增益与抑制的持续过程，输出 **CAM（当前注意记忆体）**。
- **CAM**：本 Tick 内的“核心对象集合”；既是 **结构级查存一体入口**，也是 **内源刺激生成原料**（结构级多轮次查存中分批划拨并累积转化为内源刺激）。
- **认知压**：`Δ = E_r − E_v`，`P = |Δ|`（3.3.1）。P 作为注意力资源分配的重要驱动变量。

### 1.2 工程职责（本模块要做什么）
在原型阶段，注意力模块需要承担下列职责：
1. **从状态池快照中筛选候选对象**（至少包含结构 ST；可扩展到行动节点、CFS 信号等）。
2. **形成 CAM（当前注意记忆体）**：
   - 决定“保留多少对象”（Top-N / 预算上限）。
   - 决定“给每个对象多少可消耗预算能量”（预算能量来自 SP 的能量划拨，记账式转移，不复制能量）。
3. **输出可审计报告**：为什么选它、不选它；预算从哪里来、扣了多少、剩多少；为观测台展示提供结构化字段。

### 1.3 严格不做（边界）
为避免模块边界混乱，AF 不应做：
- 不直接写 HDB（结构级/刺激级查存一体属于 HDB）。
- 不直接生成内源刺激（内源刺激来自结构级阶段按 `ρ_k` 划拨残差形成，见 3.7.2）。
- 不决策最终外显行动（行动在 Drive 竞争与行动器执行阶段完成）。
- 不把“理解文本语义”当成 AF 职责（语义结构的形成发生在查存一体与结构组织中）。

## 2. 输入输出接口设计

> 统一返回结构遵循当前工程口径：`{success, code, message, data, error, meta}`。

### 2.1 输入（MVP 阶段）
AF 在单独落地时，最小输入只需要：
- `state_pool`：状态池实例（用于 `get_state_snapshot` 与可选的 `apply_energy_update` 扣能）。
- `trace_id` / `tick_id`：用于日志串联与审计。

### 2.2 输出（MVP 阶段）
AF 输出两类东西：
1. `cam_snapshot`（`runtime_snapshot`）：
   - `top_items`：入选对象列表（至少包含 `ref_object_type == "st"` 的结构对象条目）。
   - 每个条目必须有可供结构级查存使用的预算能量字段 `er/ev`。
2. `attention_report`（观测报告）：
   - 选择依据、排序分数、预算扣能事件、预算统计（总 ER/EV/CP）等。

### 2.3 预留输入接口（为模块 5-8 联动）
后续模块完成后，AF 需要支持以下调制输入，但 **MVP 可以先不启用**：
- `focus_directives`（来自行动系统或 IESM 的“注意力行动器”输出）：
  - 典型形式：带参聚焦，参数来自 CFS 绑定的 CSA（3.10.2 示例 1）。
  - AF 在下一 Tick 对“完全匹配/高度相似”的对象给予额外增益。
- `cfs_hints`（来自认知感受模块 CFS）：
  - 违和、惊、期待、压力、繁/简、把握感等信号可作为注意力偏置。
- `emotion_modulation`（来自 EMgr/NT）：
  - 预算上限、截断阈值/增益曲线、疲劳与探索偏置、聚焦/警觉模式切换阈值等（3.9.2）。

## 3. 核心变量与公式（注意力必须对齐的“硬语义”）

### 3.1 双能量与认知压
对任意对象 `x`：
- `Δ(x) = E_r(x) − E_v(x)`
- `P(x) = |Δ(x)|`

其中：
- `Δ>0` 表示实偏多（意外输入倾向）
- `Δ<0` 表示虚偏多（过拟合/担忧倾向）

### 3.2 CAM 预算能量的“来源与性质”（必须对齐 3.7.2）
理论明确指出（3.7.2）：
- CAM 预算能量来自两部分：
  1. **AF 从 SP 划拨形成当前注意预算**
  2. **注意力对 CAM 内对象进行增益强化**
- CAM 内每个结构可以同时携带 `(E_r, E_v)` 两类预算分量，因此后续内源刺激也允许携带实/虚分量（能量类型随来源继承，不强制改写）。

### 3.3 结构级多轮次中的预算划拨（为 AF 输出字段做接口对齐）
结构级每轮匹配得到 `ρ_k ∈ [0,1]` 后，对 CAM 预算做“同倍率扣除并转移”：
- 对匹配集合 `M_k`：
  - 偏置赋能输入 `(E_r)`: `ρ_k × Σ_{s∈M_k} E_r(s)`
  - 偏置赋能输入 `(E_v)`: `ρ_k × Σ_{s∈M_k} E_v(s)`
- 对残差集合 `R_k`：
  - 内源残差片段 `I_k` 的 `(E_r)`: `ρ_k × Σ_{s∈R_k} E_r(s)`
  - 内源残差片段 `I_k` 的 `(E_v)`: `ρ_k × Σ_{s∈R_k} E_v(s)`
- 本轮后 CAM 剩余预算（同倍率衰减，保持能量序关系稳定）：
  - `E_r^{k+1}(s) = (1-ρ_k) × E_r^{k}(s)`
  - `E_v^{k+1}(s) = (1-ρ_k) × E_v^{k}(s)`

这意味着：**AF 输出的 CAM 能量字段，必须可被后续“结构级按 ρ_k 划拨”直接消费**，并能做预算记账。

## 4. AF 的默认算法（工程可落地版本）

> 理论没有强制指定一个唯一评分公式，要求的是“在预算约束下可解释地筛选/调制”。因此本节给出一个默认可审计实现，并允许由 IESM/EMgr 未来动态调参。

### 4.1 候选过滤
从 SP 快照取候选对象 `items`，先做硬过滤：
- 排除 `ref_object_type == "em"`（情景记忆不进入 CAM；与当前原型约束一致）
- 排除 `E_r + E_v <= min_total_energy` 的弱对象（默认 0，可配置）

### 4.2 优先级评分（MVP 与现有观测台保持一致）
当前 MVP 排序优先级 `U(x)` 采用：
- `U(x) = 1.25*(E_r+E_v) + 0.35*P + 0.15*salience + 1e-12*updated_at`

说明：
- 这是工程实现上的排序规则，不改变理论语义：核心仍是双能量与认知压驱动；updated_at 仅用于弱打破平局，方便观测稳定。
- 后续可通过 EMgr/IESM 引入 fatigue/recency 等调制项（3.9.2）。

### 4.3 预算划拨与扣能（“思考有代价”）
对入选对象：
- 预算能量（CAM 内可支出副本）：
  - `B_r = E_r * ratio`
  - `B_v = E_v * ratio`
- 若 `consume_energy=true`：
  - 对应从 SP 扣除 `(B_r, B_v)`（通过状态池的 `apply_energy_update` 记账式转移）
- CAM 快照内对象的 `er/ev` 字段取 `(B_r, B_v)` 作为本 Tick 可被结构级消费的预算。

## 5. 与现有原型闭环的接入点（你现在就能接入什么）

### 5.1 可立刻接入（高收益，低耦合）
当前原型已经有“注意力占位抽取”流程（observatory 中的 `_build_attention_memory_stub`）。模块 4 的 MVP 可以直接替换该位置：
- 输入：`StatePool.get_state_snapshot(top_k=None)` 的快照
- 输出：`cam_snapshot` 直接喂给 `HDB.run_structure_level_retrieval_storage(state_snapshot=cam_snapshot, top_n=...)`
- 扣能：使用 `StatePool.apply_energy_update` 实现预算能量转移（与理论一致）
- 观测：在 report 中输出 `attention_report`，前端可直接展示

### 5.2 必须预留接口（等模块 5-8 完成后再启用）
以下内容不建议在“只做注意力模块”阶段强行实现，否则会导致语义硬编码：
- 注意力行动器（依赖行动模块 Drive/行动节点竞争，见 4.2.1）
- 违和感/正确事件触发的带参聚焦（依赖 IESM 的 CFS 脚本与触发脚本，见 3.10.2 示例 1）
- “惊”导致的警觉模式切换与预算上调（依赖 CFS + EMgr/NT 调制接口，见 3.8.2、3.9.2）

## 6. 可观测性与日志（强制要求）

建议观测台至少展示：
- 候选数、入选数、Top-N、ST 数量
- 每个入选对象：`ref_object_type/ref_object_id/display`，`B_r/B_v`，`P`，`fatigue`，`recency_gain`
- 扣能事件列表（item_id、before/after、delta、reason、trace_id/tick_id）

日志分三层（error/brief/detail），并保证能用 `trace_id` 串起一次 tick 全链路。

## 7. 当前落地状态（代码对应关系）

模块 MVP 已落地于：
- `attention/main.py`：`AttentionFilter.build_cam_from_pool(...)`
- `observatory/_app.py`：`_build_attention_memory_stub()` 已替换为调用 `AttentionFilter`

调用示例：
```python
from attention import AttentionFilter
from state_pool.main import StatePool

pool = StatePool(config_override={"enable_placeholder_interfaces": False, "enable_script_broadcast": False})
af = AttentionFilter()

resp = af.build_cam_from_pool(pool, trace_id="tick_0001", tick_id="tick_0001")
cam_snapshot = resp["data"]["cam_snapshot"]
report = resp["data"]["attention_report"]
```
